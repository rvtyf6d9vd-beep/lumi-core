一、执行摘要

本计划为 Lumi 项目在 QoderWork 平台上建立一套规则驱动、子代理调度、三级合规的开发治理体系。核心架构是主调度 + Task 子代理模式：QoderWork 主会话通过 AGENTS.md 加载通用规则和任务路由，根据任务类型调度 Task 子代理注入角色专属上下文执行工作。文件系统持久化状态和跨角色变更通知。30 条有效规则编码为结构化 YAML，通过实时检查、定时扫描、里程碑门禁三重机制确保执行。

4 个角色
PM-Infra / HW-Design / HW-Ver / QA-Audit
30 条规则
12 critical / 14 high / 4 medium
3 级合规
实时 / 定时 / 里程碑门禁
分层加载
通用规则 AGENTS.md + 角色专属按需注入

4.2 角色上下文注入协议

当用户下达任务时，主会话执行以下协议：

Step 1: 任务路由

根据 AGENTS.md 中的任务路由表，判断目标角色。路由关键词：

任务关键词目标角色专属规则
需求、计划、里程碑、商务、环境、CI/CD、知识库pm-infraB-1~B-5, IT-1~IT-2
架构设计、RTL、模块设计、BSP、驱动、RTOS、工具链hw-designHD-1~HD-3
验证、测试、FPGA、回归hw-verHV-1~HV-2
审计、Sign-off、合规检查qa-auditQA-1（全部规则可见）

Step 2: Pre-Check

  • 读取 agents/{role}/context/current-state.yaml
  • 检查 agents/{role}/change-notices/ 中未处理通知
  • hw-design 专属：检查 HD-2 时间预警（active_tasks 中 started 距今 ≤ 2 天）
  • hw-ver 专属：检查是否有来自 hw-design 的设计变更通知 (HV-1)

Step 3: 子代理调度

使用 Task 子代理，prompt 中注入：

1. 角色身份声明和职责范围
2. role-prompt.md 中的角色专属规则（不含通用规则）
3. 文件权限约束（可读/可写路径）
4. Pre-Check 结果（当前状态、未处理通知摘要）
5. 具体任务描述

Step 4: Post-Verify

  • 更新 agents/{role}/context/current-state.yaml
  • 追加到 audit/instruction-log/YYYY-MM-DD.yaml
  • 如有跨角色影响，写入目标角色的 change-notices/
  • hw-design 专属：失败思路写入 knowledge/lessons-learned/ (HD-3)

4.3 任务路由与分派

以下场景使用主会话直接执行（无需子代理）：

  • 跨角色的综合分析和决策
  • 需要多轮人类确认的方案选型
  • 简单查询和状态检查

以下场景使用Task 子代理：

  • 特定角色的专项设计和编码任务
  • 验证执行和结果分析
  • QA 审计（强制独立上下文）
  • 可并行的多角色任务（如同时检查不同模块的合规性）

4.4 变更通知机制（替代 inbox）

原 inbox 机制简化为变更通知（Change Notice），保留留痕的核心价值，去掉消息队列的复杂性。

设计原则

  • 只记录跨角色影响：不是所有操作都发通知，只有影响其他角色工作范围的变更才记录
  • 追加不消费：通知文件只追加新条目，不需要"读取后标记已读"的复杂状态管理
  • 可追溯：每条通知有完整的时间戳、来源、目标、关联规则

通知格式

# agents/{target}/change-notices/YYYY-MM-DD.yaml
notices:
  - id: "CN-20260615-001"
    from: hw-design
    to: hw-ver
    type: design-change     # design-change | bug-report | spec-update | review-request
    priority: high
    timestamp: "2026-06-15T14:30:00+08:00"
    subject: "CSR 模块接口变更"
    summary: "修改了 CSR 地址映射，影响验证用例地址范围"
    related_rules: ["08", "HV-1"]
    files_changed:
      - "src/csr_map.sv"
      - "docs/design/csr-spec.md"
    action_required: "更新验证用例中的地址范围"
    status: pending          # pending | acknowledged | resolved

场景来源角色目标角色通知类型

验证发现 Bughw-verhw-designbug-report

4.5 QA 审计独立性保障

QA 审计通过 Task 子代理实现上下文级隔离：

隔离机制：
  • QA 子代理不继承主会话的对话历史——看不到设计思路、调试过程等执行细节
  • QA 子代理只读取文件系统中的产出物（代码、文档、测试结果、审计报告）
  • 通过 contextDirs 参数限定可访问目录范围
  • 基于标准化检查清单（rules/ 中的 check_criteria）逐项审计
  • 审计报告存入 audit/compliance-reports/，每条规则有检查结果和证据引用
  • 最终由人类项目负责人签署确认

六、合规检查体系（重设计）

6.2 合规评分模型

每条规则根据严重度赋予权重，合规评分按加权通过率计算：

评分公式

合规评分 = Σ(已通过规则的权重) / Σ(全部规则的权重) × 100

评分等级

等级分数区间含义颜色

WARN75% - 89%存在风险项，需要关注并限期整改■ 黄色

七、知识库建设（含淘汰机制）

知识库引入 LRU（最近最少使用）淘汰机制，替代原来的硬性 256 条上限。

条目结构（增强版）

# knowledge/index.yaml
entries:
  - id: "LL-001"
    title: "AHB burst wrap 计算需要考虑 unaligned 起始地址"
    category: lessons-learned        # lessons-learned | design-decision | best-practice
    source: "hw-design, M1 阶段"
    date: "2026-06-10"
    last_referenced: "2026-06-14"    # [新] 最近引用日期
    reference_count: 3               # [新] 引用次数
    file: "lessons-learned/LL-001.md"

淘汰策略

阈值触发条件动作
正常条目数 ≤ 200正常运行
WARNING条目数 > 200提醒合并相似条目
淘汰条目数 > 240 或 每月巡检自动归档 last_referenced 距今 > 90 天的条目
强制条目数 > 256按 last_referenced 升序归档最旧的条目直到 ≤ 240

归档机制

  • 被淘汰条目移动到 knowledge/archive/，原始 .md 文件保留
  • knowledge/archive/index.yaml 记录所有归档条目，可搜索
  • 归档条目不删除，可随时恢复到活跃知识库
  • 归档条目在 memory_search 中仍可被发现，但优先级低于活跃条目

定时巡检（cron 每周一 9:00）

1. 统计活跃条目数，检查 WARNING/淘汰/强制阈值
2. 扫描 last_referenced > 90 天的条目 → 候选归档
3. 生成巡检报告 → 通知人类确认归档操作
4. 确认后执行归档，更新 index.yaml 和 archive/index.yaml

十、风险与缓解

风险影响概率缓解措施
子代理上下文窗口不足 复杂角色的规则 + 任务描述可能超出有效上下文 中 分层加载已大幅缩减（~63%）；必要时按任务动态裁剪非相关规则
子代理无法"感知"角色切换流程 子代理不知道 Pre-Check / Post-Verify 的完整协议 高 主会话负责 Pre/Post，子代理只执行核心任务；在 prompt 中明确边界
AGENTS.md 过大消耗上下文 通用规则 + 路由表 + 协议可能超过 4 KB 中 AGENTS.md 只放摘要和路由，规则详情保留在 YAML 中按需读取
定时任务执行失败无人知晓 合规扫描静默失败，风险积累 低 定时任务结果发送到 IM 通道；连续 2 次失败触发高优先级通知
知识库淘汰误归档重要条目 90 天未引用不代表无价值 中 归档不删除，可恢复；淘汰操作需人类确认；reference_count 高的条目豁免
合规评分虚高 自动化检查覆盖面不足，manual 项默认 PASS 高 manual 检查项在评分中标记为 UNCHECKED 而非 PASS；总分分母排除未检查项
