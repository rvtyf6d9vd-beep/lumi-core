1. 执行摘要

本次对 Lumi-Core 全部 28 个 RTL 文件、16 份设计文档、11 个 TB 文件及 66 个 directed 测试执行了独立全局静态代码扫描。 分析覆盖 设计一致性、逻辑缺陷、验证完整性、需求追溯、架构合规 五个维度。

关键结论:

  • SoC 集成层断路 (3× Critical): lumi_soc_top.sv 中 trap/CLIC/NMI 信号全部硬编码为 0 或未连接，导致异常处理与中断控制器在 SoC 层面完全断路
  • 数据通路关键缺陷 (3× Critical): Cache Evict 丢失 48B 脏数据、CSR 写与 HPM 递增竞争、CDC 同步器短路
  • 流水线旁路时序 (1× Critical): E2→M 级 MUL/DIV 结果旁路存在 1-cycle 延迟窗口
  • ISA 合规缺口 (1× Critical): 13 个 Profile Mandatory 扩展中 8 个在解码层完全缺失
  • 验证覆盖不足: 35 个 Cover Groups 中仅 16 个有对应测试映射；Scoreboard 无 Reference Model 交叉验证
  • 全局参数 100% 合规: 9 项核心架构参数 (XLEN/AXI/HPM/TCM/Cache/CLIC 等) 与 platform-spec.html 完全一致

3. 逻辑缺陷清单

3.1 信号完整性 (Task 2.1)

ID严重程度文件行号问题描述修复建议
SIG-01High lumi_regfile.sv~45-70 Regfile 双写端口冲突: 当两个写端口同时写同一寄存器时，缺少仲裁逻辑。后写入的值不确定 (取决于仿真器/综合工具行为)。 添加写端口优先级仲裁: slot0 > slot1 > slot2，或添加冲突检测并 stall。
SIG-02High lumi_csr_unit.sv~120-180 MISA_VAL 不正确: MISA CSR 的硬编码值未包含所有已实现的 ISA 扩展 (如 B/Zba/Zbb 扩展位)。 根据实际已实现扩展更新 MISA_VAL，确保 MISA[25:0] 各扩展位正确反映硬件能力。
SIG-03Medium lumi_fetch.sv~280-320 BTB 写入无 ECC 保护: BTB SRAM 写入未添加 ECC 校验位，而 TCM/Cache 均有 ECC。BTB 单 bit 翻转可能导致误预测。 为 BTB 添加 SECDED ECC (与 TCM_ECC_EN 参数对齐)，或至少在 BTB 命中时添加奇偶校验。
SIG-04Low lumi_execute.sv~135 already_redirected 标志: 使用 bit 类型而非 logic，在仿真中默认为 0 但在综合中可能不确定。 改用 logic 类型或在 always_comb 块开头显式初始化。

3.2 时序与竞争条件 (Task 2.2)

ID严重程度文件行号问题描述修复建议
TIM-01Critical lumi_csr_unit.sv~90-150 CSR 写与 HPM 递增竞争: 当 CSR 写操作 (如 CSRRW 写 mhpmcounter) 与 HPM 自动递增在同一 cycle 发生时，递增逻辑后执行会覆盖写入值。数据损坏。 在 HPM 递增逻辑中添加写冲突检测: 若当前 cycle 有 CSR 写目标为 hpmcounter[n]，则跳过该 cycle 递增。
TIM-02Critical lumi_cache.sv~200-260 Cache Evict 仅发送 1/4 beat: 64B cache line 在 evict 时仅发送第一个 16B beat (AXI 128-bit = 16B/beat)，剩余 48B 脏数据丢失。需发送 4 beats。 修复 evict FSM: 添加 beat 计数器，确保发送完整 4 beats (arlen=3) 后才完成 evict。
TIM-03Critical lumi_axi_wrapper.sv~50-100 CDC 同步器短路: 跨时钟域信号 (clk_core ↔ clk_bus) 直接连线无同步器。亚稳态传播风险极高。 实现标准 2-FF CDC 同步器 (单 bit) 和 DMUX/异步 FIFO (多 bit)。参考 LUMI-CLK-001 需求。
TIM-04High lumi_debug_module.sv~80-140 Debug CDC 处理不当: JTAG TCK 域 (低速 ~10MHz) 与系统 clk_core 域 (高速 ~100MHz+) 之间仅使用单级触发器同步。 实现 TCK→sys 的 2-FF 同步 + DMUI 握手协议 (JTAG 写请求 → sys 响应 → JTAG 确认)。

3.3 状态机完整性 (Task 2.3)

ID严重程度文件FSM问题描述修复建议
FSM-01Medium lumi_fetch.sv取指 FSM (5状态) case 语句有 default 分支 ✓，但 default 仅执行空操作。异常状态跳转未定义。 default 分支应跳转到 ST_IDLE 并置错误标志。
FSM-02Medium lumi_dft_ctrl.svMBIST FSM MBIST 测试 FSM 缺少 default 状态处理。综合工具可能推断出锁存器。 添加 default: state_next = ST_IDLE; 安全回退。
FSM-03Medium lumi_lockstep.sv锁步比较 FSM 缺少 default 分支。锁步错误后 FSM 可能进入不确定状态。 添加 default 回退到 IDLE 并触发安全中断。
FSM-04Medium lumi_power_mgmt.sv功耗管理 FSM ST_SLEEP 状态不可达 — 没有任何状态转移指向 ST_SLEEP。该状态为死代码。 实现 ST_ACTIVE → ST_SLEEP 转移条件 (如 WFI 指令触发)，或移除未使用的状态定义。
FSM-05Low lumi_cache.svCache 控制器 FSM FSM 有 default 分支 ✓，复位进入 ST_IDLE ✓。但 evict 子状态机的状态编码未使用 one-hot。 改用 one-hot 编码以提高综合质量和时序。

3.4 资源冲突与死代码 (Task 2.4)

ID严重程度文件问题描述修复建议
RES-01Critical lumi_core_top.sv E1→M MUL/DIV 旁路时序: E2 级产生的 MUL/DIV 结果在传递到 M 级时，旁路网络未正确对齐 1-cycle 延迟，导致 M 级读取到 E1 级的旧操作数而非 E2 结果。 在旁路 MUX 中添加 E2→M 路径，使用 E2 级输出寄存器值而非 E1 级组合逻辑值。
RES-02High lumi_fpu.sv FPU 运算为 passthrough: FPU 模块当前仅实现数据透传 (输入直接连接到输出)，无实际浮点运算逻辑。所有 F/D 扩展指令返回未修改的操作数。 按 fpu-vector.html 实现 FADD/FMUL/FDIV/FSQRT/FCVT 运算单元。当前阶段可添加 "FPU not implemented" 异常。
RES-03High lumi_vector.sv Vector SIMD 硬编码为加法: 向量模块仅实现 SIMD 加法运算，乘法/除法/比较/转换等操作均返回加法结果。 按 fpu-vector.html §4 实现完整 SIMD 运算集，或对未实现操作触发 illegal instruction 异常。
RES-04Low lumi_decode_issue.sv RV32C 扩展保留字段: 压缩指令展开后，inst_raw[15:0] 保留原始 16-bit 指令。但 32-bit 非压缩指令仅存低 16-bit，高 16-bit 信息丢失。 对 32-bit 指令，inst_raw 应保存完整 32-bit 或明确文档说明仅用于调试。
