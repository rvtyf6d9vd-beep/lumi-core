一、执行摘要

本计划为 Lumi 项目在 Cowork 平台上建立一套规则驱动、角色切换、自动化合规的开发治理体系。核心架构是双引擎模式：Cowork 主会话（当前默认模型）作为主引擎，通过读取角色上下文文件实现「角色切换」，负责所有复杂推理任务；qodercli --yolo 作为轻量引擎，负责格式检查、文件整理、日志汇总等简单可控任务。文件系统实现「状态持久化」和「角色间通信」。所有 29 条规则被编码为结构化 YAML 数据，通过 CLAUDE.md 指令、合规脚本和 CI 管道三重机制确保执行。

双引擎架构
Cowork 主会话 + qodercli 轻量委派
6 个角色上下文
PM / HW-Arch / HW-Ver / SW-Arch / AI-Infra / QA-Audit
29 条规则 · 3 层检查
任务级 / 日构建级 / 里程碑级

四、Agent 架构（Cowork 双引擎）

4.1 双引擎执行模型

Cowork 适配架构的核心是双引擎分工：Cowork 主会话（默认模型）处理所有需要推理的复杂任务，qodercli --yolo（轻量模型）处理简单可控的常规操作。

4.2 角色上下文切换机制

当主会话以某个角色工作时，需要加载该角色的完整上下文。以下是 CLAUDE.md 中定义的切换流程：

# Lumi-core/CLAUDE.md 中的角色切换指令

## 角色切换协议

当用户要求以某个角色工作，或者任务明确属于某个角色时，执行：

1. 读取 agents/{role}/role-prompt.md 获取该角色的规则绑定和行为约束
2. 读取 agents/{role}/context/current-state.yaml 获取该角色的当前状态
3. 读取 agents/{role}/inbox/ 目录下的所有未处理消息
4. 在执行任务前，按照 role-prompt.md 中的 Pre-Check 流程检查前置条件
5. 执行任务
6. 任务完成后，更新 agents/{role}/context/current-state.yaml
7. 如果任务产出需要通知其他角色，写入目标角色的 inbox/ 目录
8. 写入 audit/instruction-log/YYYY-MM-DD.yaml

## 可用角色
| 角色标识 | 显示名称 | 提示词文件 |
|---------|---------|-----------|
| pm | 项目经理 | agents/pm/role-prompt.md |
| hw-arch | 硬件架构 | agents/hw-arch/role-prompt.md |
| hw-ver | 硬件验证 | agents/hw-ver/role-prompt.md |
| sw-arch | 软件架构 | agents/sw-arch/role-prompt.md |
| ai-infra | AI 基础设施 | agents/ai-infra/role-prompt.md |
| qa-audit | QA 审计 | agents/qa-audit/role-prompt.md |

4.3 qodercli 委派协议

简单可控的常规任务通过 qodercli --yolo 在主机 Shell 中执行。qodercli 使用其默认轻量模型，运行在独立进程中，与 Cowork 主会话天然隔离。

委派原则（已有实践规则）：简单可控任务走 qodercli --yolo，复杂推理任务自己做。判断标准是「任务的输入输出是否确定」和「是否需要多步推理或人类确认」。

qodercli 委派示例

# 示例 1: 批量检查文档日期标记（规则 03）
# 主会话通过 host_bash 工具委派给 qodercli
qodercli --yolo -p "检查 docs/ 目录下所有 .html 文件是否包含 2026 年的日期标记，
列出缺少日期标记的文件。只输出文件列表，不做修改。" \
--cwd /path/to/Lumi-core

# 示例 2: 汇总 inbox 未读消息
qodercli --yolo -p "读取 agents/*/inbox/ 目录下所有 status: unread 的 yaml 文件，
按角色分组汇总消息数量和优先级，输出为简洁表格。" \
--cwd /path/to/Lumi-core

# 示例 3: 运行合规检查脚本并格式化输出
qodercli --yolo -p "运行 python rules/scripts/check_compliance.py，
将输出格式化为 YAML 写入 audit/compliance-reports/2026-06-10.yaml" \
--cwd /path/to/Lumi-core

# 示例 4: 知识库条目计数（IT-1: ≤256 条）
qodercli --yolo -p "统计 knowledge/index.yaml 中的条目数量，
如果超过 240 条则在输出中标记 WARNING。" \
--cwd /path/to/Lumi-core

# 示例 5: git 工作区状态检查（规则 13）
qodercli --yolo -p "检查 git status，如果有未提交的变更则列出文件清单，
确认是否有遗漏的 Task-ID 关联。" \
--cwd /path/to/Lumi-core

# 里程碑 M1 门禁：主会话 + qodercli 串行协作

# 步骤 1: qodercli 执行自动化检查（轻量模型）
qodercli --yolo -p "依次运行以下检查脚本并汇总结果：
1. python rules/scripts/check_doc_completeness.py --milestone M1

4.6 QA 审计的独立性保障

QA-1 规则：计划、执行过程、结果都必须由第三方 Agent 审计。在单会话架构中，QA 审计由同一个 Cowork 主会话执行，存在「自己审自己」的风险。v2.1 通过 qodercli 进程隔离 + 标准化检查清单 + 人类最终确认 三重保障确保审计独立性。

QA 独立性保障措施

1
进程级上下文隔离 — QA 审计通过 qodercli --yolo 在独立进程中执行。qodercli 进程不继承主会话的对话历史和角色执行细节（设计思路、调试过程等），天然实现上下文隔离。QA 只看到文件系统中的产出物（文档、代码、测试结果），不看到产出过程。
2
标准化检查清单 — QA 审计基于 rules/ 中的结构化规则和可量化检查标准（如回归通过率 100%、需求覆盖率 100%），而非主观判断。qodercli 的 prompt 中内嵌完整的检查清单模板，确保审计覆盖面不遗漏。
3
审计报告可追溯 — qodercli 产出的审计报告存入 audit/compliance-reports/，包含每条规则的检查结果和证据引用，供人类项目负责人复核。
4
人类最终签署 — QA 审计报告需要人类项目负责人最终确认，作为门禁通过的最后一道关卡。任何审计结论均不自动生效。
为什么 qodercli 隔离优于子 Agent 隔离？ 子 Agent 由主会话派生，理论上主会话可以在 prompt 中注入执行上下文。而 qodercli 是独立进程，只能通过文件系统获取输入，物理上无法接触主会话的对话历史——这是比 prompt 级隔离更强的进程级隔离。

4.7 模型分层调用策略（双引擎）

对应规则 B-7：用最聪明的模型制定计划，用操作损耗最低的方式完成常规操作。v2.1 采用双引擎分层：Cowork 主会话（默认模型）处理需要深度推理的任务，qodercli（其默认模型）处理简单可控任务。
引擎调用方式任务层级场景示例

成本优化效果：qodercli 使用其默认的轻量模型，Token 成本远低于主会话模型。将 50%+ 的常规操作委派给 qodercli，可显著降低 B-2 规则要求的 Token 成本预算压力。

规则管理系统的设计与 v1.0 基本一致——它本质上是文件和脚本，不依赖多 Agent 基础设施。以下保留核心设计并标注 Cowork 适配点。

5.1 规则文件格式与元数据

所有规则存储为结构化 YAML，每条包含 id、title、category、severity、applicable_roles、applicable_phases、description、check_type、check_criteria 等字段。格式与 v1.0 相同，此处不再重复。

Cowork 适配点：v1.0 中 sync_agents.py 用于更新各 Agent 的 CLAUDE.md。在 v2.0 中替换为 role_prompt_builder.py：根据 role-mapping.yaml 动态生成每个角色的 role-prompt.md，确保角色提示词始终与最新规则同步。 
#!/usr/bin/env python3
"""role_prompt_builder.py — 根据角色映射生成角色提示词文件"""

import yaml
from pathlib import Path

def build_role_prompt(project_root, role_id):
    root = Path(project_root)

    # 1. 加载角色映射
    with open(root / "rules/role-mapping.yaml") as f:
        mapping = yaml.safe_load(f)
    role = mapping["roles"][role_id]

    # 2. 加载该角色适用的规则全文
    applicable_rules = []
    for rule_file in (root / "rules").glob("**/*.yaml"):
        if rule_file.name in ("schema.yaml", "role-mapping.yaml"):
            continue
        with open(rule_file) as f:
            data = yaml.safe_load(f)
        for r in data.get("rules", []):
            if r["id"] in role["rules"] or "ALL" in role["rules"]:
                applicable_rules.append(r)

    # 3. 组装提示词
    prompt = f"# {role['display_name']} — 角色提示词\n\n"
    prompt += f"## 角色定义\n你正在以 **{role['display_name']}** 角色执行任务。\n\n"
    prompt += "## 适用规则\n"
    for r in applicable_rules:
        prompt += f"### [{r['id']}] {r['title']}\n{r['description']}\n\n"
    prompt += f"## 文件权限\n"
    prompt += f"- 可读: {role['permissions']['read']}\n"
    prompt += f"- 可写: {role['permissions']['write']}\n"
    # ... Pre-Check / Post-Verify 流程

    # 4. 写入角色提示词文件
    out = root / f"agents/{role_id}/role-prompt.md"
    out.write_text(prompt)
    print(f"已更新 {out}")

6.2 里程碑门禁机制

1

九、版本管理与 GitHub 工作流

版本管理策略（分支策略、提交规范、PR 审查流程、CI 工作流）与 v1.0 完全相同。这些机制基于 GitHub 和 CI/CD，不受 Cowork Agent 模型的影响。

唯一的 Cowork 适配点：Cowork 主会话通过 Bash 工具执行 git 操作，遵循 CLAUDE.md 中定义的提交规范（commit message 格式、Task-ID 关联、Rules-Checked 标注）。

十、知识库建设

知识库的结构、知识条目格式、规模控制机制（≤256 条）与 v1.0 完全相同。

Cowork 协同点：Cowork 自身的 memory/ 系统与 Lumi 项目的 knowledge/ 目录分工明确。memory/ 存储跨项目的用户偏好和工作方式（如「exec summary 不要 hedge」），knowledge/ 存储项目内的技术经验和设计决策。两者互不重叠。

十三、风险与缓解

风险影响概率缓解措施Cowork 特有?
角色切换时遗忘加载上下文 高中 CLAUDE.md 中的角色切换协议强制要求读取 role-prompt.md 和 current-state.yaml。如果忘记读取，Post-Verify 会发现缺少状态更新而报错。 是
qodercli 输出不符合预期格式 中中 qodercli prompt 中明确要求输出结构化结果（YAML 格式）并指定输出文件路径。主会话执行后检查输出文件是否存在且格式正确，异常时重试或手动处理。 是
文件信箱消息被遗漏 中中 角色切换协议的 Pre-Check 步骤强制检查 inbox/ 目录。Cowork 定时任务每日扫描所有 inbox/ 汇总未读消息。 是
QA 审计独立性不足（自己审自己） 高低 qodercli 进程级隔离（物理上无法接触主会话对话历史）+ 标准化检查清单 + 人类最终确认。比 v2.0 的 prompt 级隔离更强。 是
顶层 CLAUDE.md 过长导致上下文压力 中低 CLAUDE.md 只放协议和路由表，规则全文在 role-prompt.md 中按需加载。保持 CLAUDE.md < 200 行。 是
Agent 执行时未严格遵守权限约束 中低 role-prompt.md 软约束 + CODEOWNERS PR 硬约束 + QA 审计覆盖。 —
知识库增长超过 256 条 低中 Cowork 月度定时任务检查条目数 + 240 条预警 + 里程碑后强制精简。 —
Token 成本超支 高中 模型分层调用 + Cowork 定时任务周报 + 15% 预警阈值。 —
规则更新后角色提示词未同步 中低 规则更新后立即运行 role_prompt_builder.py；Cowork 从文件系统读取，天然同步。 —
