## 1. 评估目标

本报告通过 gem5 性能模型对当前流水线设计进行定量参数扫描，评估以下四个维度的优化空间：

1. **性能进一步提升** — 是否能通过参数调优获得更高 IPC
2. **流水线时序拉平** — 是否能通过分级调整综合到更高主频
3. **总线接口时序优化** — 在总线接口时延 ≈ 30% 总接口时延约束下的优化路径
4. **PPA 优化** — 面积/功耗/性能三者权衡的理论最优配置

> **方法论说明**: gem5 提供 cycle-accurate 的性能数据，可直接量化参数变化对 IPC 的影响。但 gem5 **不直接建模** RTL 时序路径（时钟频率）、芯片面积和功耗。本报告采用 "gem5 性能数据 + 架构推理" 的方法，对时序和 PPA 进行半定量估算，结论为方向性参考而非精确预测。

## 2. 当前配置基线

### 2.1 流水线结构

```
Fetch1 → Fetch2 → Decode1 → Decode2 → Issue → Execute → Memory → Writeback
 (BTB)    (I$)    (pre-dec) (reg-read) (disp)  (ALU/Br)  (D$)     (wb)

## 4. 四维度定量分析

### 4.1 性能进一步提升

#### 4.1.1 发现：当前配置已接近顺序执行 ILP 天花板

| 配置 | IPC | 相对基线 |
|------|-----|----------|
| 基线 (4 ALU) | 1.915 | — |
| 最优组合 (6 ALU) | 1.918 | +0.16% |
| 理论峰值 (3-issue) | 3.000 | +56.7% |
| **实际 ILP 利用率** | **63.8%** | — |

**核心结论**: 在顺序执行架构下，CoreMark 的指令级并行性 (ILP) 已被充分挖掘。从 4 ALU 增至 6 ALU 仅获得 +0.2% 性能提升，说明 FU 停顿已被 skip-stalled 机制有效吸收。

#### 4.1.2 各参数性能敏感度排序

| 参数 | 敏感度 | ΔIPC/ΔParam | 说明 |
|------|--------|-------------|------|
| **D-Cache 延迟** | **极高** | -11.9% / +1 cycle | 最关键时序参数 |
| **Load 推测** | **高** | -6.9% / (on→off) | 第二关键机制 |
| ALU 数量 | 低 | +0.2% / +2 ALU | 已被 skip-stalled 吸收 |
| 缓冲深度 | 极低/负 | -0.5% / +1x | 更大缓冲反而有害 |
| 分支延迟 | 低 | -0.2% / +1 cycle | LTAGE 误判率极低 |
| Skip-stalled | 零 | 0% / (on→off) | CoreMark 无收益 |

#### 4.1.3 性能提升路径

**路径 A: 架构级 (需 RTL 重设计)**
- 乱序执行 (OoO): 可提升 IPC 至 2.5-2.8 (+30-46%)，但面积/功耗代价大
- 宽发射 (4-wide): 顺序执行下收益有限，预计 +5-8%
- SMT: 不适用于实时确定性场景

**路径 B: 微架构级 (可在当前框架内优化)**
- 数据预取增强: 当前 Stride Prefetcher degree=2，可尝试 degree=4
- 寄存器重命名: 消除假 WAR/WAW 依赖，预计 +3-5%
- 分支目标缓存 (BTB) 优化: 当前 8192 entries，BTB miss 集中在 DirectCond (84.8%)，可考虑 BTB 预解码

**路径 C: 编译器级 (零硬件代价)**
- 指令调度: GCC -O2 已有基础调度，可尝试 LTO (Link-Time Optimization)
- 软件流水化: 对 CoreMark 紧密循环有效
- 寄存器分配优化: 减少 scoreboard 停顿

### 4.2 流水线时序拉平 (更高主频)

#### 4.2.1 当前各级关键路径分析

```
级          关键路径逻辑                    估计相对延迟
─────────────────────────────────────────────────
Fetch1      BTB 查找 + I$ tag 比较          ████░  (中)
Fetch2      I$ data 读取                    ███░░  (低)
Decode1     预解码 (RVC 识别)               ██░░░  (低)
Decode2     全解码 + 寄存器堆读取           █████  (高) ← 关键路径
Issue       调度逻辑 (FU 选择)              ███░░  (中)
Execute     ALU 运算 + 分支解析             ████░  (中-高)
Memory      D$ 访问 (tag+data)              █████  (高) ← 关键路径
Writeback   寄存器堆写入                    ███░░  (中)
```

#### 4.2.2 流水线分割方案

**方案 1: Decode2 分割 (Decode2a + Decode2b)**

```
Decode2a: 全指令解码
Decode2b: 寄存器堆读取
```

| 指标 | 估算 |
|------|------|
| 新流水线级数 | 9 级 |
| 性能代价 | ~+1 cycle decode latency → 估算 -0.3% 至 -0.5% |
| 时序收益 | 关键路径缩短 ~40% (解码和寄存器读取分离) |
| 主频提升空间 | 估算 +30-50% |
| **净吞吐量** | **+29% 至 +49%** (主频提升远大于性能损失) |
| 面积代价 | +1 级流水寄存器 (~微量) |

**方案 2: Memory 分割 (Mem1 + Mem2)**

```
Mem1: D$ tag 比较
Mem2: D$ data 读取
```

| 指标 | 估算 |
|------|------|
| 新流水线级数 | 9 级 |
| 性能代价 | D-Cache 延迟 1→2 周期 → **-11.9%** (实测) |
| 时序收益 | 关键路径缩短 ~50% |
| 主频提升空间 | 估算 +50-80% |
| **净吞吐量** | **+33% 至 +59%** (1.5/1.119=1.34 至 1.8/1.119=1.61) |
| 面积代价 | +1 级流水寄存器 |

**方案 3: Execute 分割 (Exec1 + Exec2)**

```
Exec1: FU 运算
Exec2: 结果选择 + 旁路网络
```

| 指标 | 估算 |
|------|------|
| 新流水线级数 | 9 级 |
| 性能代价 | 分支反馈延迟 1→2 周期 → **-0.2%** (实测 C2) |
| 时序收益 | ALU 关键路径缩短 ~30% |
| 主频提升空间 | 估算 +20-30% |
| **净吞吐量** | **+20% 至 +30%** |
| 面积代价 | +1 级流水寄存器 + 旁路网络复杂化 |

#### 4.2.3 组合方案评估

| 组合 | 新级数 | 性能损失 | 所需主频提升 (盈亏平衡) | 推荐主频目标 | 净收益估算 |
|------|--------|----------|------------------------|-------------|-----------|
| Decode2 分割 | 9 级 | -0.5% | +0.5% | 1.3-1.5 GHz | **+29-49%** |
| Execute 分割 | 9 级 | -0.2% | +0.2% | 1.2-1.3 GHz | **+20-30%** |
| D$ 分割 | 9 级 | -11.9% | +13.5% | 1.5-1.8 GHz | **+33-61%** |
| Decode2 + Execute | 10 级 | -0.7% | +0.7% | 1.4-1.6 GHz | **+39-59%** |
| **全部分割** | **11 级** | **-12.6%** | **+14.4%** | **1.6-2.0 GHz** | **+41-77%** |

> **关键结论**: 流水线时序拉平的盈亏平衡点很低（Decode/Execute 分割仅需 +0.7% 主频即可盈利）。即使是最昂贵的 D$ 分割 (+11.9%)，只要主频提升超过 13.5% 即可获得净收益。**推荐优先执行 Decode2 和 Execute 分割**，性价比最高。

#### 4.3.2 总线接口时延约束分析

用户约束: **总线接口时延 ≈ 30% 总接口时延**

当前状态: 总线在性能上几乎无影响 (0.15% 利用率)，但在 RTL 时序上，AXI 接口逻辑 (地址解码、仲裁、响应聚合) 可能在关键路径上占有显著比例。

**估算模型**:

```
总接口时延 = 核心接口时延 + 总线接口时延
总线接口时延 ≈ 30% × 总接口时延
核心接口时延 ≈ 70% × 总接口时延
```

**优化路径**:

| 方案 | 描述 | 时序收益 | 性能代价 | 面积代价 |
|------|------|----------|----------|----------|
| **A: AXI 寄存器切片** | 在 AXI master/slave 边界插入流水寄存器 | 高 (隔离总线与核心) | ~0% (总线利用率 0.15%) | 低 (+N 个寄存器) |
| **B: 总线宽度提升** | 128-bit → 256-bit | 中 (减少 mux 逻辑) | ~0% (miss rate 极低) | 中 (+数据线) |
| **C: Harvard 总线分离** | 指令/数据独立总线 | 中 (消除共享仲裁) | ~0% | 中 (+1 条总线) |
| **D: TCM 直连** | DTCM 绕过总线直连 LSU | 高 (实时数据零延迟) | 正向 (TCM 命中 0 周期) | 低 (已有 TCM 模块) |

**推荐方案 A (AXI 寄存器切片)**:

- 在 CPU 核心和 AXI 总线之间插入一级寄存器
- AXI 的 5 个通道 (AW/W/B/AR/R) 各加一级寄存器
- 时序收益: 总线接口路径增加 1 周期延迟，但从关键路径中移除
- 性能代价: 缓存 miss 时增加 1 周期 → 130 D$ miss × 1 cycle = 130 周期 → 0.002% 性能损失 (可忽略)
- 面积代价: ~128 bit × 5 通道 × 2 (req+resp) = ~1280 个触发器
- **关键优势**: 将总线接口时延从关键路径移至独立流水级，核心时序不受总线制约

**估算效果**:
```
优化前: 关键路径 = 核心逻辑 + 总线接口逻辑 (30%)
优化后: 关键路径 = 核心逻辑 (70%)  ← 总线接口移出
         总线接口 = 独立 1 周期
理论主频提升 = 1 / 0.70 = 1.43x → +43%
```

## 5. 综合建议

### 5.1 优化路线图

```
优先级排序 (性价比 = 收益 / 实施难度):

### 5.2 推荐实施路径

**阶段 1: 时序优化 (目标: +30-50% 主频)**

- **条件: 阶段 1 主频提升已达 30%+，物理设计确认 D$ 是新关键路径**

### 5.3 gem5 模型能力边界

| 分析维度 | gem5 能力 | 精度 | 补充方法 |
|----------|-----------|------|----------|
| **性能 (IPC/CM/MHz)** | ✅ 完全支持 | 精确 (cycle-accurate) | — |
| **流水线时序 (关键路径)** | ⚠️ 间接 | 半定量 | 需 RTL 综合报告确认 |
| **总线接口时序** | ⚠️ 间接 | 半定量 | 需 STA 报告确认 30% 约束 |
| **面积** | ❌ 不支持 | — | 需 CACTI/综合工具 |
| **功耗** | ❌ 不支持 | — | 需 PrimeTime-PX/RTL 仿真 |

> **结论**: gem5 性能模型可以提供**不准确的定量分析**——性能维度精确，时序维度半定量 (通过参数扫描估算上下界)，面积和功耗维度仅能提供架构级推理。完整 PPA 评估需结合 RTL 综合流程。
