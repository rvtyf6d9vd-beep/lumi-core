1. 概述与范围

1.1 模块职责

L1 Cache 模块为 Lumi-Core 提供指令缓存 (I-Cache) 和数据缓存 (D-Cache)，加速对外部 DDR 内存的访问：

  • I-Cache：F1 级发出取指请求，F2 级返回指令数据。对 TCM 地址透明（TCM 路径优先）
  • D-Cache：M 级执行 Load/Store 访问。采用推测性延迟预测 (assume hit, latency=1)，miss 时回滚
  • ECC 保护：可选的 SECDED 校验，保护 Cache data RAM 和 tag RAM
  • RVWMO 支持：实现 RISC-V Weak Memory Ordering 模型的 FENCE 语义

2.2 Cache 操作流程

2.2.1 I-Cache 读取 (F1→F2)

// I-Cache 取指流程
function icache_fetch(pc):
    // 1. 先检查 TCM 地址空间
    if pma_check(pc).region == TCM:
        return TCM_PATH  // 走 tcm.html 路径

    // 2. Cache 查找
    set_idx  = pc[12:6]     // 7 bit 索引
    tag      = pc[31:13]    // 19 bit tag
    offset   = pc[5:0]      // 6 bit 偏移

    // 3. 并行比较 4 个 way 的 tag
    for way in 0..3:
        if tag_array[set_idx][way].valid and tag_array[set_idx][way].tag == tag:
            // HIT: 返回 cacheline 中 offset 对应的指令数据
            data = data_array[set_idx][way][offset]
            update_plru(set_idx, way)
            return {data, ICACHE_HIT}

    // 4. MISS: 发起 refill
    return ICACHE_MISS  // 触发 refill 状态机

2.2.2 D-Cache Load (M 级)

// D-Cache Load 流程 (推测性延迟预测)
function dcache_load(addr, size):
    // 推测性假设: D-Cache hit, latency=1 (GEM5 executeLoadAssumedLatency=1)
    // Issue 级已假设 hit 并发射后续指令

    // 1. TCM 路由检查
    if pma_check(addr).region == TCM:
        return TCM_PATH  // 走 TCM 路径

    // 2. PMA: cacheable 检查
    if not pma_check(addr).cacheable:
        return axi_direct_read(addr)  // 非缓存访问, 直接 AXI

    // 3. Cache 查找
    set_idx = addr[12:6]
    tag     = addr[31:13]
    for way in 0..3:
        if tag_array[set_idx][way].valid and tag_array[set_idx][way].tag == tag:
            data = data_array[set_idx][way][addr[5:0]]
            update_plru(set_idx, way)
            return {extract(data, addr, size), DCACHE_HIT}

    // 4. MISS: 需要 refill
    // 推测性已假设 hit, 此处发现 miss → pipeline rollback
    pipeline_rollback()  // flush D/I/E1/E2/M 级中推测执行的指令
    initiate_refill(addr, set_idx)
    return DCACHE_MISS

2.2.3 D-Cache Store (M 级)

// D-Cache Store 流程 (Write-back + Write-allocate)
function dcache_store(addr, data, byte_enable):
    // 1. TCM 路由
    if pma_check(addr).region == TCM:
        return TCM_PATH

    // 2. Non-cacheable → AXI 直写
    if not pma_check(addr).cacheable:
        return axi_direct_write(addr, data, byte_enable)

    // 3. Cache 查找
    set_idx = addr[12:6]
    tag     = addr[31:13]
    for way in 0..3:
        if tag_array[set_idx][way].valid and tag_array[set_idx][way].tag == tag:
            // HIT: 写入 data array + 置 dirty bit
            merge_write(data_array[set_idx][way], addr[5:0], data, byte_enable)
            tag_array[set_idx][way].dirty = 1
            update_plru(set_idx, way)
            return STORE_OK

    // 4. MISS + Write-allocate: 先 refill 再写入
    initiate_refill(addr, set_idx)
    // refill 完成后重试 store
    return STORE_PENDING

2.3 替换策略 审查约束: 替换策略

2.3.1 Pseudo-LRU (PLRU) 选型理由

策略硬件开销命中率复杂度评估
True LRU6 bit/set (4-way: C(4,2)=6)最优高 (全排列比较)面积大, 时序差
Pseudo-LRU (PLRU)3 bit/set (4-way 二叉树)接近 LRU (<1%差距)低 (树形 bit 翻转)选中 ✓
Round-Robin2 bit/set差 (无时间局部性感知)最低命中率不足
RandomLFSR差低不确定性高

2.3.2 PLRU 算法 (4-way 二叉树)

// PLRU 二叉树结构 (4-way)
//
//        b0
//       /  \
//     b1    b2
//    / \   / \
//  W0  W1 W2  W3
//
// bit=0 → 指向左子树 (最近访问方向)
// bit=1 → 指向右子树
// 替换: 从根沿 bit 方向走到叶子, 即为 LRU 近似 way

// 查找替换目标
function plru_victim(set_idx):
    b0 = plru_bits[set_idx][0]
    b1 = plru_bits[set_idx][1]
    b2 = plru_bits[set_idx][2]
    if b0 == 0:
        if b1 == 0: return WAY_0
        else:       return WAY_1
    else:
        if b2 == 0: return WAY_2
        else:       return WAY_3

// 命中后更新 PLRU bits (将命中 way 标记为"最近使用")
function update_plru(set_idx, hit_way):
    // 将路径上的 bit 翻转, 使后续 victim 远离 hit way
    case hit_way:
        WAY_0: plru_bits[set_idx] = {1, 1, b2}   // b0→1, b1→1
        WAY_1: plru_bits[set_idx] = {1, 0, b2}   // b0→1, b1→0
        WAY_2: plru_bits[set_idx] = {0, b1, 1}   // b0→0, b2→1
        WAY_3: plru_bits[set_idx] = {0, b1, 0}   // b0→0, b2→0
审查约束响应 — 替换策略:
  • 选型: PLRU (Pseudo-LRU), 4-way 仅需 3 bit/set, 比 True LRU 节省 50% 存储
  • 性能影响: PLRU vs True LRU 命中率差距 <1% (文献与 GEM5 仿真验证)
  • 面积影响: 32KB I-Cache 128 sets × 3 bit = 384 bit = 48 Byte PLRU 存储, 可忽略
  • 时序影响: PLRU victim 选择为 2 级 MUX (2 gate delay), 不在关键路径上
  • 验证策略: Cover Group cg_replacement_policy 覆盖全部 PLRU 状态转移

2.4 Cache Refill 状态机

// Refill FSM (I-Cache 和 D-Cache 共享逻辑)
//
// IDLE → REFILL_REQ → REFILL_WAIT → REFILL_RECV → REFILL_DONE → IDLE
//
// 单次 refill: 64 Byte cacheline = 4 beat × 128bit AXI burst (AWLEN=3)
// 或: 8 beat × 64bit, 取决于 AXI 总线宽度

enum RefillState { IDLE, REFILL_REQ, REFILL_WAIT, REFILL_RECV, REFILL_DONE, EVICT }

function refill_fsm(addr, set_idx):
    state = IDLE

    // IDLE → REFILL_REQ: 发起 AXI AR 请求
    state = REFILL_REQ
    axi_araddr  = addr[31:6] << 6  // cacheline 对齐
    axi_arlen   = 3                  // 4 beat burst (128bit bus)
    axi_arsize  = 4                  // 16 byte per beat
    axi_arvalid = 1

    // REFILL_WAIT: 等待 AXI ARREADY
    wait_until(axi_arready)

    // REFILL_RECV: 接收 4 beat 数据
    state = REFILL_RECV
    for beat in 0..3:
        wait_until(axi_rvalid)
        refill_buffer[beat] = axi_rdata

    // EVICT (D-Cache only): 如果 victim 行 dirty, 先写回
    victim_way = plru_victim(set_idx)
    if tag_array[set_idx][victim_way].dirty:
        state = EVICT
        axi_writeback(set_idx, victim_way)  // AXI AW/W burst

    // REFILL_DONE: 写入新行
    state = REFILL_DONE
    tag_array[set_idx][victim_way] = {valid:1, dirty:0, tag:addr[31:13]}
    data_array[set_idx][victim_way] = refill_buffer
    reset_plru(set_idx, victim_way)  // 新行设为 MRU
    state = IDLE

2.5 ECC 保护 LUMI-MEM-007

2.5.1 ECC 覆盖范围

保护对象数据位宽ECC 位宽描述
Data RAM (per line)512 bit (64 Byte)10 bitSEC(512+10, 522) 或分段保护
Tag RAM (per way)19 bit tag + 1 valid + 1 dirty6 bit保护 tag 完整性
实际实现考量: 64 Byte cacheline 的 SEC 需要 10 bit ECC，硬件开销较大。实际可采用分段 ECC：每 32 Byte 独立 ECC (8 bit)，降低解码延迟并支持部分写入。具体分段策略在 RTL 设计阶段确定。

2.5.2 ECC 错误处理

错误类型位置处理
Single-bit CEData RAM自动纠正 + 回写, 无流水线影响
Double-bit DEDData RAM触发 Load/Store Access Fault (mcause=5/7)
Tag ECC 错误Tag RAM视为 cache miss + invalidate, 触发 refill

2.6 FENCE 与内存序 LUMI-MEM-008

2.6.1 RVWMO 模型实现

RISC-V 采用 RVWMO (RISC-V Weak Memory Ordering) 内存序模型，L1 Cache 需支持以下 FENCE 语义：

指令语义硬件实现
FENCE通用内存屏障排空 Store Buffer, 等待所有 pending Store 完成
FENCE.I指令流同步Invalidate I-Cache 全部行 + 排空 pipeline
FENCE.TSOTSO 序 (RVWMO 子集)等效 FENCE rw,rw (全序约束)

2.6.2 FENCE 执行伪代码

function execute_fence(fence_type):
    case fence_type:
        FENCE:
            drain_store_buffer()        // 所有 pending store 写回
            wait_pending_refill()       // 等待进行中的 refill
            // 不 flush I-Cache (FENCE 不影响指令缓存)

        FENCE_I:
            drain_store_buffer()
            icache_invalidate_all()     // 全部 I-Cache 行 invalidate
            pipeline_flush()            // flush F2/D/I/E1/E2/M/W
            // 下一条取指将从新地址重新填充 I-Cache

        FENCE_TSO:
            // TSO: 等效 FENCE rw,rw
            drain_store_buffer()
            wait_pending_refill()

2.7 Cache-TCM 路由

// M 级内存访问路由决策
function memory_route(addr, op_type):
    pma = pma_check(addr)  // 来自 pma-axi.html

    if pma.region == TCM:
        // TCM 路径: 直接访问 TCM (参见 tcm.html)
        return route_to_tcm(addr, op_type)

    elif pma.cacheable:
        // Cache 路径: 走 I-Cache 或 D-Cache
        if op_type == FETCH:
            return icache_fetch(addr)
        elif op_type == LOAD:
            return dcache_load(addr)
        else:
            return dcache_store(addr)

    else:
        // Non-cacheable: 直接 AXI 访问
        if op_type == LOAD:
            return axi_direct_read(addr)
        else:
            return axi_direct_write(addr)

4. 数据通路与控制

4.1 I-Cache 数据通路

// I-Cache 数据通路
//
// F1: PC → [Tag RAM 并行读出 4 way] → [Comparator × 4]
//                                         │
//                              ┌──────────┼──────────┐
//                              ▼          ▼          ▼
//                          Way0 Hit   Way1 Hit   Way2 Hit   Way3 Hit
//                              │          │          │          │
//                              ▼          ▼          ▼          ▼
//                          [Data RAM MUX (4→1)] → 指令数据 → F2
//
// 关键路径: Tag Comparator (19bit XOR + NOR tree) ~3 gate delays
// 1GHz @ 28HPC: 1000ps budget, Tag compare ~150ps, 裕量充足

4.2 D-Cache 数据通路 (Load)

//                             │
