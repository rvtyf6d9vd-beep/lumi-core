1. 概述与约定

1.1 文档目的

本文档集中定义 Lumi-Core 全部模块间的交互接口，包括信号清单、握手协议、时序约束、数据不变式和异常处理。每份 LLD 文档的 §3 接口约束表应与本文档对应章节保持一致。

2. 流水线级间接口

本节定义 8 级流水线 (F1→F2→D→I→E1→E2→M→W) 的级间传递接口。当前 RTL 使用 scalar 端口（独立信号线）在 lumi_core_top.sv 中进行级间连接，每级信号在顶层 always_ff 块中通过流水线寄存器捕获或组合逻辑直传。控制信号包括 valid/ready/stall/flush，数据包为 inst_pkt_t 结构体数组。

  • PC 对齐: inst_pkt[0].pc[1:0] 必须为 2'b00（32-bit 指令）或 2'b10（16-bit 压缩指令，偶数地址）

Stall/Flush 语义

  • stall=1: F1 级保持当前 PC 不变，不向 F2 推送新数据；F2 级保持现有数据不更新
  • flush=1: F1 级丢弃当前预测路径，下一周期从 trap_pc（trap）或 e1_br_target（误预测）重新取指；F2 级清除当前缓存数据

数据不变式

  • 指令边界正确性: pd_inst_valid[i]=1 当且仅当 pd_inst[i] 是完整指令（非跨 cache line 片段）
  • PC 递增规则: 对于连续指令流，pd_inst_pc[i+1] = pd_inst_pc[i] + (pd_inst_compressed[i] ? 2 : 4)
  • valid 高后数据不可变: f2_valid=1 期间，所有 pd_inst* 信号保持稳定直到 ready=1 且 clk_core 上升沿
  • 预解码一致性: 若 pd_inst_compressed[i]=1，则 pd_inst[i] 必须是 32-bit 展开后的指令，pd_inst_raw[i] 存原始 16-bit 编码

Stall/Flush 语义

  • stall=1 (dec_stall): D 级解码器忙或 DIB 满，F2 级保持当前预解码结果不更新；F1 级同步暂停
  • flush=1: F2 级丢弃当前缓存的 ICache 数据和预解码结果；D 级 DIB FIFO 不清空（已入队指令继续处理）
  • ready 生成: ready = dib_not_full，当 DIB FIFO 剩余空间 ≥1 时拉高

三发射语义说明

D→I 接口支持最多三发射（ISSUE_WIDTH=3），inst_pkt[0:2] 和 inst_valid[2:0] 的含义如下：

  • inst_pkt[0]: 程序序最老的指令（PC 最小），优先发射到对应 FU
  • inst_pkt[1]: 程序序中间的指令
  • inst_pkt[2]: 程序序最新的指令（PC 最大）
  • inst_valid[0]=1: slot[0] 有有效指令待发射
  • inst_valid[1]=1: slot[1] 有有效指令待发射（仅当 slot[0] 也有效时才可能）
  • inst_valid[2]=1: slot[2] 有有效指令待发射（仅当 slot[0:1] 均有效时才可能）
  • 发射组合: 合法组合为 inst_valid ∈ {3'b000, 3'b001, 3'b011, 3'b111}（连续有效，不允许空洞）

数据不变式

  • inst_pkt_t 字段完整性: 对于 inst_valid[i]=1 的指令包，所有字段（pc, inst, rd, rs1, rs2, funct3, funct7, imm, fu_type, fu_id, privilege）必须已正确填充
  • 寄存器地址合法性: rd/rs1/rs2 ∈ [0, 31]，rd=0 表示不写寄存器（如 store/branch）
  • fu_type 与指令匹配: fu_type 必须与指令操作码解码结果一致（ALU/MUL/DIV/MEM/FP/VEC/BRANCH/MISC）
  • valid 高后数据不可变: valid=1 期间，inst_pkt[0:2] 和 inst_valid[2:0] 保持稳定直到 ready=1 且 clk_core 上升沿

Stall/Flush 语义

  • stall=1 (dec_stall): I 级目标 FU 忙（fu_busy[fu_type]=1）或除法器忙（div_busy=1），D 级暂停出队，DIB FIFO 保持当前深度
  • flush=1: D 级清空 DIB FIFO 中所有未发射指令；I 级丢弃当前发射批次；flush 信号由 e1_mispredict || trap_request 驱动
  • ready 生成: ready = !fu_busy[inst_pkt[0].fu_type] && !div_busy（至少最老指令可发射）

数据不变式

  • 操作数就绪: 对于 e1_valid_r[i]=1 的指令，e1_rs1_data_r[i] 和 e1_rs2_data_r[i] 必须已是最终值（寄存器堆读出或旁路命中后的结果）
  • 旁路优先级: 若旁路网络命中（bp_rs1_hit[i]=1），则使用旁路数据 bp_rs1_data[i] 而非寄存器堆读出值
  • 分支预测状态: e1_pred_taken_r 和 e1_pred_target_r 仅对 branch 类型指令有效，其他指令可忽略

Stall/Flush 语义

  • stall=1: I→E1 流水线寄存器保持当前值不更新；E1 级继续执行当前指令但结果不向 M 级传递
  • flush=1: I→E1 流水线寄存器全部清零（e1_valid_r='0）；flush 由 e1_mispredict || trap_request 触发
  • 分支气泡: 当 E1 级有 branch 指令时（e1_has_branch=1），I→E1 寄存器暂停捕获（插入气泡），防止推测指令进入 E1

数据不变式

  • E2 结果旁路: 对于 MUL/DIV 指令，m_result_r[i] 使用 E2 级输出（e2_mul_result/e2_div_result）而非 E1 级 ALU 结果
  • 误预测分支保留: 误预测分支指令本身（如 JAL）必须被捕获到 M 级（用于写 ra），但其后的推测指令被杀死
  • mem_busy 保护: 当 mem_busy=1 时，E1→M 寄存器保持不更新，防止双 MEM 串行化期间重复捕获
  • e1_mem_* 再寄存: e1_mem_addr / e1_mem_we / e1_mem_wdata 虽为 E1 级组合输出，但在 lumi_memory_stage 模块入口被再寄存一拍，与 m_inst_r / m_valid_r 对齐后用于 DCache/TCM 访问

Stall/Flush 语义

  • stall=1 (mem_busy): M 级 Load/Store 访问未完成，E1→M 寄存器暂停捕获新数据
  • flush=1 (trap): M 级输入寄存器全部清零，异常指令不继续流向 W 级
  • flush=1 (mispredict): 误预测分支指令保留，其后的推测指令清零

  • 门控检查: mie_gate=0 时 CLIC FSM 保持 ST_IDLE，不响应任何中断
  • 抢占安全: irq_preempt=1 仅在 ST_PENDING 状态且新中断级别 > latch_level 时产生

时序说明

CLIC FSM 流程: ST_IDLE → ST_DETECT (确认 pending) → ST_PRIORITY (锁存 ID/level) → ST_LATENCY (等待 HW_LAT cycles) → ST_PENDING (发出 irq_valid) → ST_ACK (等待 irq_ack) → ST_IDLE。抢占时从 ST_PENDING 回到 ST_PRIORITY。

SoC 实际行为: 由于 irq_ack 固定为 1'b0，CLIC FSM 在进入 ST_ACK 状态后无法正常返回 ST_IDLE（state_next 在 ST_ACK 时无条件跳转到 ST_IDLE，因此 FSM 实际上会返回 IDLE，但无法完成完整的中断应答握手）。中断请求 irq_valid 仅在 ST_PENDING 期间有效，irq_request 传送到 lumi_exception 模块参与 trap 决策。

Claim/Complete 握手说明

PLIC 标准 claim/complete 流程如下：

  1. Claim: CPU 读 claim/complete 寄存器（MMIO 地址 0x104 + h*8），PLIC 返回当前最高优先级中断 ID 并锁存为 claimed_src，同时清除 pending 位
  2. 处理: CPU 执行中断处理程序
  3. Complete: CPU 写 claim/complete 寄存器（写入中断 ID），PLIC 清除 claimed_valid，允许该中断源再次 pending
当前状态: PLIC MMIO 接口全部接零（plic_addr=12'h0, plic_wdata=32'h0, plic_we=1'b0），irq_claim=1'b0。因此 claim/complete 握手不可用，PLIC 输出 meip 虽有内部逻辑但悬空未连接到 CLIC。CLIC 的 meip_in 直接接 1'b0，意味着 PLIC 外部中断路径完全未打通。

Mispredict 特殊处理说明

E1→M 寄存器在 mispredict 时的特殊行为:
  • 误预测分支保留: 当 e1_mispredict=1 时，E1→M 寄存器不会完全清零。误预测分支指令本身（如 JAL ra）会被保留到 M 级（m_valid_r[i]=1 仅对 fu_type==FU_BRANCH 的指令），以确保其寄存器写入（如 ra）在 W 级完成
  • 推测指令杀死: 同批次的其他非分支指令被清零（m_valid_r[i]=0），这些是基于错误预测路径获取的指令
  • 无条件保留: ERR-022 修复后移除了 !m_valid_r[0] 检查，无条件保留误预测分支，防止 JAL 的 ra 写入丢失
  • post_mispredict_bubble: 若误预测分支写 rd≠0，额外插入 1 周期气泡（post_mispredict_bubble_r），让 W-level bypass 提供正确的 ra 值给紧随的依赖指令（如 RET 读 ra）
Trap flush 行为: trap_request=1 时 E1→M 寄存器完全清零（包括所有指令），异常指令不会继续流向 W 级。这与 mispredict 的保留策略不同。
