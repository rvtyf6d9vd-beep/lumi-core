1. 概述与范围

1.1 模块职责

Debug & Trace 模块提供 Lumi-Core 的在线调试和指令跟踪能力：

  • Debug Module：JTAG 访问的调试模块，支持断点、单步、寄存器/内存读写
  • Trigger Module：硬件断点/观察点 (mtrigger/strigger)
  • Abstract Commands：通过 DMI 执行调试命令 (无需 halt 即可读寄存器)
  • System Bus Access：通过 Debug Module 访问系统总线 (不经过 Hart)
  • Trace Encoder：RISC-V Nexus Trace 指令流压缩编码
  • Trace AXI4 写出：Trace 数据通过 AXI4 写出到外部存储

2. 功能描述

2.1 Debug Module 架构

2.1.1 DMI 接口

Debug Module 通过 DMI (Debug Module Interface) 与外部 JTAG 调试器通信：

信号方向位宽描述

2.2 Hart Halt/Resume 接口 DEC-007-1

2.2.1 Halt 机制

// Debug halt 请求流程
function debug_halt_request():
    // 1. Debug Module 设置 dmcontrol.haltreq = 1
    // 2. Hart 在当前指令完成后进入 halt 状态
    halt_pending = 1

function debug_halt_check():  // W 级检查
    if halt_pending and current_instruction_committed:
        // 精确 halt: 当前指令已提交, 后续指令不取
        hart_state = HALTED
        halt_pending = 0
        dmstatus.allhalted = 1
        // 流水线 flush (F2/D/I/E1/E2/M)
        pipeline_flush()
        // Hart 等待 Debug Module 命令

// Debug resume 流程
function debug_resume():
    // dmcontrol.resumereq = 1
    hart_state = RUNNING
    dmstatus.allrunning = 1
    // 从 halt PC 继续执行
    PC = halted_pc
审查约束响应 — DEC-007-1 (Debug halt/resume 接口):
  • 接口定义: debug_halt_req (In) + debug_halted (Out) + debug_resume_req (In)
  • 同步机制: halt 在 W 级检测，保证精确断点 (当前指令已提交)
  • 与中断互斥: halt pending 时不接受新中断 (mstatus.MIE 隐式置 0)
  • 与 NMI 关系: NMI 可打断 halt (NMI 优先级 > halt)
  • 时序: halt 请求到 Hart 实际 halted: 0~8 cycle (取决于当前指令类型)

2.3 Abstract Command 执行

2.3.1 Access Register (cmdtype=0)

// Abstract Command: Access Register
function abstract_access_register(cmd):
    regno  = cmd[31:16]  // 寄存器编号
    write  = cmd[15]     // 0=read, 1=write
    size   = cmd[14:12]  // 访问大小 (2=32bit, 3=64bit)
    transfer = cmd[17]   // 是否传输数据
    postexec = cmd[18]   // 执行后运行 progbuf

    if transfer:
        if write:
            // 从 data0 写入寄存器
            register_write(regno, data0)
        else:
            // 从寄存器读到 data0
            data0 = register_read(regno)

    if postexec:
        // 执行 progbuf 中的指令
        execute_progbuf()

2.3.2 Quick Access (cmdtype=1)

Quick Access 允许不 halt Hart 即可执行 progbuf 中的指令序列，用于低侵入性内存访问。

2.6 调试安全 D-016

安全等级能力控制方式
Level 0: 完全锁定禁止所有调试操作debug_lock 引脚或 dmcontrol
Level 1: 仅断点允许 halt/单步/断点, 禁止内存读写CSR 配置
Level 2: 完全开放允许所有调试操作默认
审查约束响应 — D-016 (Debug 安全性): 已在 §2.6 定义 3 级安全等级。锁定状态下禁止 Abstract Command、System Bus Access 和 Program Buffer 执行。解锁不可通过软件绕过，需硬件复位或密钥认证。Cover Group cg_debug_security 覆盖 3 级安全模式。

4. 数据通路与控制

4.1 Debug Module 时钟域

Debug Module 工作在 tck 时钟域 (JTAG)，与核心 clk_core 异步：

// 跨时钟域处理
//
// tck 域 (JTAG, 10~50MHz)          clk_core 域 (1GHz)
// ┌──────────────┐                  ┌──────────────┐
// │ DMI Request  │ ── 2-FF Sync ──→│ DM Register  │
// │ DMI Response │ ←── Async FIFO ─│ Access Logic │
// └──────────────┘                  └──────────────┘
//
// CDC 策略:
// - DMI req/resp: 异步握手 (req_valid → req_ack → resp_valid)
// - halt/resume: 脉冲同步 (1-cycle pulse 跨域传递)
// - data 传输: Async FIFO (深度 4)

4.2 硬件断点匹配

// Trigger Module: 地址匹配断点
function trigger_match(pc, mem_addr, mem_data):
    for i in 0..NUM_TRIGGERS-1:
        t = mcontrol[i]
        if not t.dmode or current_privilege == M:
            // 检查 match 类型
            if t.match == EQUAL and t.data_reg == pc:
                return TRIGGER_HIT(i)
            if t.match == NAPOT and napot_match(pc, t.data_reg):
                return TRIGGER_HIT(i)
    return TRIGGER_MISS
