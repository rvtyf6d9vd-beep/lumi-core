1. 概述与范围

1.1 模块职责

TCM 模块为 Lumi-Core 提供紧耦合、确定性延迟的片上存储，覆盖 M 级 (Memory stage) 的数据访问和 F1/F2 级的取指访问：

  • ITCM：指令紧耦合内存，用于确定性取指（Boot 和关键代码路径）
  • DTCM0/DTCM1：数据紧耦合内存，奇偶双区域，支持并行访问
  • ECC 引擎：可选的 SECDED (Single-bit Error Correction, Double-bit Error Detection) 保护
  • DMA 从接口：外部 DMA 控制器可通过从接口读写 TCM 内容

2. 功能描述

2.1 TCM 区域划分

2.1.1 地址映射 LUMI-MEM-003

区域起始地址结束地址默认大小访问类型描述
ITCM0x0000_00000x0000_7FFF32KBR/X (取指+读)代码 TCM，用于 Boot 和关键代码
DTCM00x0001_00000x0001_7FFF32KBR/W (数据)数据 TCM 偶区
DTCM10x0002_00000x0002_7FFF32KBR/W (数据)数据 TCM 奇区
奇偶双区设计: DTCM0/DTCM1 按地址奇偶位分体，当两条 LSU 分别访问 DTCM0 和 DTCM1 时可同周期并行完成，避免体冲突。三发射模式下 2xLSU 同周期最多 2 条 Load/Store，双区域最大化并行概率。

每个 TCM 区域容量独立可配，必须为 8KB 的 2 的幂次倍数 LUMI-MEM-002：

2.2 访问延迟模型 LUMI-MEM-004

2.2.1 零拍模式 (Zero-latency)

默认模式。TCM 访问在 M 级 (Memory stage) 内完成，不引入额外 stall：

// 零拍模式时序 — Load 指令
// F1  F2  D   I   E1  E2  [M]   W
//                          ^
//                     TCM 读出, 数据同 M 级可用
// Load→Use stall: 1 cycle (推测性延迟预测, executeLoadAssumedLatency=1)

// 零拍模式时序 — Store 指令
// F1  F2  D   I   E1  E2  [M]   W
//                          ^
//                     TCM 写入, M 级提交

2.2.2 一拍模式 (One-latency)

TCM 访问需要额外 1 cycle，M 级 stall 等待数据返回：

// 一拍模式时序 — Load 指令
// F1  F2  D   I   E1  E2  [M]  [M+1]  W
//                          ^     ^
//                     TCM REQ  TCM 数据返回
// Load→Use stall: 2 cycle (1 cycle 推测 + 1 cycle TCM 额外延迟)
// 流水线: M 级 stall 1 cycle, 前级连锁 stall
审查约束 — TCM 延迟切换: 0/1 拍模式通过配置寄存器 tcm_latency_cfg 切换。切换必须在流水线空闲时执行 (FENCE 或 pipeline drain)，否则正在 M 级的 TCM 访问会产生时序错误。切换流程：
  1. 软件写入 tcm_latency_cfg CSR
  2. 硬件检测配置变更，插入 pipeline drain (等待 M/W 级排空)
  3. 排空完成后新延迟模式生效
  4. 后续 TCM 访问使用新延迟
切换开销: ≤4 cycle (drain 2~3 条 in-flight 指令)。

2.2.3 延迟切换伪代码

// TCM 延迟切换控制逻辑
register tcm_latency_cfg : bit = 0  // 0 = zero-latency, 1 = one-latency

function tcm_access(addr, op_type, data_in):
    // 地址解码: 判断是否命中 TCM 区域
    region = decode_tcm_region(addr)
    if region == TCM_NONE:
        return TCM_MISS  // 非 TCM 地址, 走 Cache/AXI 路径

    // 延迟模式选择
    if tcm_latency_cfg == 0:
        // 零拍: M 级内完成
        if op_type == LOAD:
            return tcm_sram_read(region, addr)  // 同 cycle 返回
        else:
            tcm_sram_write(region, addr, data_in)
            return STORE_OK
    else:
        // 一拍: M 级 stall 1 cycle
        pipeline_stall(1)  // stall M 级及所有前级
        if op_type == LOAD:
            return tcm_sram_read(region, addr)  // 下一 cycle 返回
        else:
            tcm_sram_write(region, addr, data_in)
            return STORE_OK

function tcm_latency_switch(new_latency):
    // 切换流程: 必须 drain 流水线
    tcm_latency_cfg = new_latency
    pipeline_drain()  // 等待 M/W 级排空, 约 2~3 cycle
    // 新延迟模式生效

2.3 ECC 保护引擎 LUMI-MEM-005

2.3.1 ECC 编码方案

采用 SECDED (Single Error Correction, Double Error Detection) 方案，基于 Hamming Code + 整体奇偶位：

参数值描述
数据位宽32 bit单字 ECC
ECC 位宽7 bit (6 Hamming + 1 parity)SEC(32,39) + 整体奇偶
存储开销+21.9% (7/32)每 32bit 数据增加 7bit ECC
单 bit 翻转可检测并纠正 (CE)自动纠正, 无流水线影响
双 bit 翻转可检测不可纠正 (DED)触发 NMI 或 bus error 异常
多 bit 翻转可能漏检>2 bit 超出 SECDED 能力

2.3.2 ECC 读写流程

// ECC 编码 (写入时)
function ecc_encode(data32):
    hamming_bits = compute_hamming_parity(data32)  // 6 bit
    overall_parity = parity(data32 ++ hamming_bits) // 1 bit
    return {data32, hamming_bits, overall_parity}   // 39 bit

// ECC 解码 (读出时)
function ecc_decode(stored39):
    data32     = stored39[38:7]
    hamming6   = stored39[6:1]
    overall_p  = stored39[0]

    syndrome = compute_syndrome(data32, hamming6)
    recomputed_p = parity(stored39[38:1])

    if syndrome == 0 and (recomputed_p XOR overall_p) == 0:
        return {data32, NO_ERROR}           // 无错误
    elif syndrome != 0 and (recomputed_p XOR overall_p) == 1:
        corrected = correct_single_bit(data32, syndrome)
        return {corrected, CORRECTED_ERROR}  // 单 bit 纠正
    elif syndrome != 0 and (recomputed_p XOR overall_p) == 0:
        return {data32, UNCORRECTABLE_ERROR} // 双 bit 检测, 不可纠正
    else:
        return {data32, UNCORRECTABLE_ERROR} // 多位错误

2.3.3 ECC 错误处理

错误类型检测处理动作流水线影响
无错误syndrome=0, parity match正常返回数据0 cycle
单 bit CEsyndrome≠0, parity mismatch自动纠正 + 回写修正值 + 计数+10 cycle (透明纠正)
双 bit DEDsyndrome≠0, parity match触发 bus_error_exception + NMITrap (flush 流水线)
Scrubbing: ECC 纠正后的数据自动回写 (write-back) 到 TCM，修复单 bit 翻转。同时 ecc_ce_count CSR 计数器递增，供软件监控内存健康状态。DED 错误触发 Store/AMO Access Fault (mcause=7) 异常。

2.4 DMA 从接口 LUMI-IF-003

DMA 控制器通过专用从接口访问 TCM，允许在不经过核心流水线的情况下加载/卸载 TCM 内容：

2.4.1 DMA 访问协议

特性规格
接口类型AXI4 Slave (独立于主总线)
数据位宽32bit (与 TCM 字宽匹配)
Burst 支持INCR burst, 最大 16 beat
访问延迟1 cycle (DMA 侧, 独立于核心 TCM 延迟配置)
仲裁策略核心优先 — 核心 M 级访问 TCM 时 DMA 请求被 stall

2.4.2 核心-DMA 仲裁

// TCM 端口仲裁逻辑
// TCM SRAM 为双端口: Port A = 核心, Port B = DMA
// 单端口 SRAM 时使用时分复用

function tcm_arbitrate(core_req, dma_req):
    if core_req.valid and dma_req.valid:
        // 核心优先
        grant_port(core_req)
        dma_req.stall()
    elif core_req.valid:
        grant_port(core_req)
    elif dma_req.valid:
        grant_port(dma_req)
    else:
        idle()

// DMA Burst 处理
function dma_burst_handler(addr, len, direction):
    for i in range(len):
        if core_access_pending():
            wait(1)  // 核心优先, DMA stall
        if direction == READ:
            data = tcm_read(addr + i*4)
            axi_slave_send(data)
        else:
            data = axi_slave_recv()
            tcm_write(addr + i*4, data)
DMA 一致性: DMA 写入 TCM 后，核心的取指/Load 可立即读到新值 (TCM 无 Cache 层)。但如果核心正在执行 TCM 中的代码，DMA 写入 ITCM 可能导致取指不一致。软件应在 DMA 加载 ITCM 后执行 FENCE.I 指令刷新指令缓冲。

4.3 取指路径 (ITCM)

F1 级送出 PC 地址后，若地址落在 ITCM 范围，F2 级直接从 ITCM 获取指令数据：

// ITCM 取指路径
function fetch_from_itcm(pc):
    if pc >= ITCM_BASE and pc < ITCM_BASE + itcm_size:
        // 取 64bit (2 条 32bit 指令或 4 条 16bit 压缩指令)
        // 对齐到 16bit 边界
        aligned_addr = pc & ~0x3
        insn_data = itcm_read(aligned_addr, 8)  // 读 8 bytes
        return insn_data
    else:
        return CACHE_PATH  // 走 I-Cache 路径
