4.4 死锁/活锁分析

  • 死锁防护: FSM 无循环依赖。ST_LATENCY 有明确计数器 (latency_cnt)，保证 HW_LAT cycle 后必然转移到 ST_PENDING。ST_PENDING 等待外部 irq_ack，由 CPU trap 逻辑保证响应。
  • 活锁防护: ST_PENDING 的抢占回到 ST_PRIORITY 时，新中断级别必须严格大于当前 latch_level，因此不会发生同级抢占的无限循环。每次抢占必然提升中断级别，最多 256 次后达到最高级别。
  • 中断丢失防护: 在 ST_DETECT 状态二次确认 pending，避免毛刺触发虚假中断。抢占时不回 ST_IDLE 而是回 ST_PRIORITY，保证高优先级中断不被遗漏。

5.1.1 优先级仲裁逻辑

仲裁采用组合逻辑并行扫描，分四轮依次比较：

  1. 本地中断 (CLIC): 从 NUM_INT-1 到 0 逆序扫描，条件: irq_local[i] && level > mintthresh && level > current_mil
  2. 外部中断 (PLIC): meip_in 固定 level=0xFF, ID=11
  3. 定时器中断 (CLINT): mtip_in 固定 level=0xF0, ID=7
  4. 软件中断 (CLINT): msip_in 固定 level=0x80, ID=3

同级别中断按 高 ID 优先 原则仲裁。每轮比较条件: level > highest_level || (level == highest_level && id > highest_id)

7.1 参数约束关系

  • NUM_INT 必须 ≤ 240 (CLIC 规范最大中断数)，且 > 0
  • HW_LAT 必须 ≥ 0；$clog2(HW_LAT) 决定 latency_cnt 位宽，当 HW_LAT=0 时 $clog2(0) 行为依赖综合工具
  • NUM_INT 与 irq_id 位宽 (8-bit) 的关系: NUM_INT ≤ 256
  • 全局参数依赖: lumi_pkg::CLIC_INT_INPUTS, lumi_pkg::CLIC_HW_LATENCY

8.3 关键算法

// CLIC 优先级仲裁伪代码 (always_comb)
highest_pending = 0; highest_id = 0; highest_level = 0;

// 轮1: 本地中断扫描 (逆序, 高 ID 优先)
for i = NUM_INT-1 downto 0:
    if irq_local[i] AND irq_level[i] > mintthresh AND irq_level[i] > current_mil:
        if !highest_pending OR irq_level[i] > highest_level
           OR (irq_level[i] == highest_level AND i > highest_id):
            highest_pending = 1; highest_id = i; highest_level = irq_level[i]

// 轮2: PLIC 外部中断 (level=0xFF, id=11)
if meip_in AND 0xFF > mintthresh AND 0xFF > current_mil:
    if !highest_pending OR 0xFF > highest_level:
        highest_pending = 1; highest_id = 11; highest_level = 0xFF

// 轮3: 定时器中断 (level=0xF0, id=7)
if mtip_in AND 0xF0 > mintthresh AND 0xF0 > current_mil:
    同上比较逻辑

// 轮4: 软件中断 (level=0x80, id=3)
if msip_in AND 0x80 > mintthresh AND 0x80 > current_mil:
    同上比较逻辑
