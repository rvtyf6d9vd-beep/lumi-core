1. 概述与范围

1.1 模块职责

多核 & 锁步模块管理 Lumi-Core 的多核拓扑和功能安全锁步：

  • 多核互联：1~4 核可配置拓扑，核间通过共享内存 + 原子操作通信
  • PLIC 多核路由：PLIC 支持多 Hart 中断分发
  • CLIC 独立性：每个 Hart 拥有独立 CLIC 实例
  • 缓存一致性：双核硬件一致性 / 四核软件维护 + CMO
  • 双核锁步：面向 ASIL-D 的指令级比较锁步模式
  • 锁步比较器：W 级比较 (基线) + M 级扩展预留

2.1.2 核间通信

// 核间通信机制
//
// 1. 共享内存: 多个 Hart 通过 AXI 总线访问同一物理地址
//    - 缓存一致性保证数据正确 (硬件/软件)
//    - 原子操作 (AMO) 提供互斥原语
//
// 2. 原子操作 (A 扩展):
//    - LR (Load Reserved): 加载并设置 reservation
//    - SC (Store Conditional): 条件存储, reservation 有效时成功
//    - AMO (Atomic Memory Op): swap/add/and/or/xor/min/max
//
// 3. 软件中断 (MSI): Hart 间通过写 CLINT msip 寄存器触发
//    - hart_i 写 hart_j 的 msip → hart_j 收到软件中断
//
// 4. Fence: SFENCE.VMA / CMO 指令用于多核 TLB/Cache 同步
//    - 四核配置: 软件负责 CMO 同步 (IPI + fence)

2.1.3 多核中断路由

组件多核行为
PLIC每个中断源有 per-Hart enable 位; Claim/Complete 独立; 优先级 per-Hart threshold
CLIC每个 Hart 独立 CLIC 实例; 本地中断不跨核; 通过 PLIC 或 MSI 实现跨核通知
CLINTPer-Hart msip (软件中断) + mtip (定时器); 共享 mtime 计数器

2.2 双核锁步 LUMI-MCORE-002 LUMI-SAFE-002

2.2.1 锁步执行模型

// 双核锁步 (Dual Lock-Step) 执行模型
//
// Hart 0 (Leader) 和 Hart 1 (Checker) 执行完全相同的指令流:
// - 两核从相同 PC 开始执行
// - 每个指令周期后比较结果
// - 不一致时触发 Lock-Step Error
//
// 前提条件:
// - 两核共享同一时钟域 (clk_core)
// - 两核接收相同的中断/异常信号
// - 两核的 TCM/Cache 配置完全相同
//
// ┌─────────────┐     ┌─────────────┐
// │  Hart 0     │     │  Hart 1     │
// │  (Leader)   │     │  (Checker)  │
// │             │     │             │
// │  W 级输出 ──┼──┐  │  W 级输出 ──┼──┐
// └─────────────┘  │  └─────────────┘  │
//                  ▼                    ▼
//          ┌──────────────────────────────┐
//          │   Lock-Step Comparator       │
//          │   比较: PC + Reg + Mem Write │
//          └──────────────┬───────────────┘
//                         │
//                         ▼ mismatch
//                  ┌──────────────┐
//                  │ Error Handler│
//                  │ NMI/Reset/Log│
//                  └──────────────┘

2.2.2 比较粒度 DEC-007-3 D-011

// 锁步比较器接入点 (DEC-007-3)
//
// === 基线方案: W 级比较 (MiniMax 建议) ===
//
// 在 W 级比较以下信号:
// 1. PC[31:0]           — 当前指令地址
// 2. rd_addr[4:0]       — 目标寄存器号
// 3. rd_data[31:0]      — 写入寄存器的数据
// 4. mem_write_valid    — 是否有内存写操作
// 5. mem_write_addr[31:0] — 内存写地址
// 6. mem_write_data[31:0] — 内存写数据
// 7. mem_write_size[2:0]  — 内存写大小
// 8. exception_valid    — 是否触发异常
// 9. exception_cause[31:0] — 异常原因
//
// W 级比较优点: 实现简单, 信号稳定 (已提交的指令)
// W 级比较局限: Store Buffer 异步写入可能遗漏 (Store 在 M 级发出, W 级确认)

// === 扩展预留: M 级比较 (Deepseek 建议) ===
//
// 在 M 级额外比较:
// 10. mem_write_data_m[31:0] — M 级内存写数据 (Store Buffer 入口)
// 11. mem_write_addr_m[31:0] — M 级内存写地址
//
// M 级比较需 Store Buffer 同步:
// - Hart 0 和 Hart 1 的 Store Buffer 必须同步步进
// - 复杂度较高, M2-S1 先实现 W 级基线, 预留 M 级接口
审查约束响应 — D-011 (锁步比较粒度分歧):
  • 分歧: MiniMax 建议仅 W 级 (简单)；Deepseek 建议 W+M 双级 (全面)
  • 融合策略: M2-S1 基线采用 W 级比较，预留 M 级扩展接口 (mem_write_*_m 信号)
  • 升级路径: 若 W 级比较无法覆盖 Store Buffer 故障模式，后续版本升级到 W+M 双级
  • 面积影响: W 级比较器约 +2K gates; W+M 双级约 +5K gates + Store Buffer 同步逻辑
  • 验证: 基线覆盖 W 级全部 9 个比较信号; 扩展验证覆盖 Store Buffer 故障注入
审查约束响应 — DEC-007-3 (锁步比较器接入点):
  • 接入点: W 级提交信号 (commit_valid) 触发比较，确保比较的是已提交指令
  • 信号列表: PC + rd_addr + rd_data + mem_write_* + exception_* (9 组信号)
  • 时序: 比较结果在 W 级下一 cycle 可用 (组合逻辑比较 + 1 FF 同步)
  • M 级预留: 预留 mem_write_data_m / mem_write_addr_m 接口信号位

2.2.3 锁步比较器实现

// Lock-Step Comparator (W 级基线)
//
// module lockstep_comparator (
//     input  wire        clk,
//     input  wire        rst_n,
//     // Hart 0 (Leader) W 级信号
//     input  wire        h0_commit_valid,
//     input  wire [31:0] h0_pc,
//     input  wire [4:0]  h0_rd_addr,
//     input  wire [31:0] h0_rd_data,
//     input  wire        h0_mem_wr_valid,
//     input  wire [31:0] h0_mem_wr_addr,
//     input  wire [31:0] h0_mem_wr_data,
//     input  wire [2:0]  h0_mem_wr_size,
//     input  wire        h0_exc_valid,
//     input  wire [31:0] h0_exc_cause,
//     // Hart 1 (Checker) W 级信号
//     input  wire        h1_commit_valid,
//     input  wire [31:0] h1_pc,
//     input  wire [4:0]  h1_rd_addr,
//     input  wire [31:0] h1_rd_data,
//     input  wire        h1_mem_wr_valid,
//     input  wire [31:0] h1_mem_wr_addr,
//     input  wire [31:0] h1_mem_wr_data,
//     input  wire [2:0]  h1_mem_wr_size,
//     input  wire        h1_exc_valid,
//     input  wire [31:0] h1_exc_cause,
//     // 比较结果
//     output reg         lockstep_error,
//     output reg  [3:0]  error_code
// );
//
// 比较逻辑:
//   commit_mismatch  = h0_commit_valid ^ h1_commit_valid
//   pc_mismatch      = h0_commit_valid & (h0_pc != h1_pc)
//   rd_mismatch      = h0_commit_valid & (h0_rd_addr != h1_rd_addr | h0_rd_data != h1_rd_data)
//   mem_mismatch     = h0_mem_wr_valid & (h0_mem_wr_addr != h1_mem_wr_addr |
//                                           h0_mem_wr_data != h1_mem_wr_data |
//                                           h0_mem_wr_size != h1_mem_wr_size)
//   exc_mismatch     = h0_exc_valid & (h0_exc_valid != h1_exc_valid | h0_exc_cause != h1_exc_cause)
//
//   lockstep_error = commit_mismatch | pc_mismatch | rd_mismatch | mem_mismatch | exc_mismatch
//   error_code = {commit_mismatch, pc_mismatch, rd_mismatch, mem_mismatch}

2.2.4 锁步错误处理

处理策略触发方式行为
NMI (默认)lockstep_nmi 信号两核同时收到 NMI, 进入 Smrnmi 处理流程
系统复位lockstep_reset 信号两核同时复位, 从复位向量重新开始
日志记录lockstep_log CSR记录 mismatch 详情 (PC/error_code/timestamp) 供诊断

错误处理策略可通过 lockstep_error_mode CSR 配置，默认 NMI。

6.2 关键验证场景

  • 锁步故障注入: 在 Hart1 W 级注入 PC/RD/MEM 偏差，验证比较器检测
  • 锁步中断同步: 中断到达时两核同时进入中断处理，比较器持续工作
  • 多核 AMO 竞争: 两个 Hart 同时对同一地址执行 LR/SC
  • 锁步错误恢复: NMI 处理后重新同步两核执行
  • Store Buffer 场景: 验证 W 级比较对 Store Buffer 故障的覆盖度
