1. 概述与范围

1.1 模块职责

CLIC & PLIC 模块实现 Lumi-Core 的两层中断架构，负责中断信号的聚合、仲裁和分发：

  • PLIC (Platform-Level)：平台级中断控制器，聚合外部中断源 (32~960)，执行优先级仲裁 (7 级)，多 Hart 路由
  • CLIC (Core-Local)：ISA 扩展层中断控制器，per-hart 实例，管理本地中断 + PLIC 传入的外部中断，支持硬件向量中断和优先级抢占

2.2 PLIC 详细设计

2.2.1 中断源管理

参数规格
中断源数32~960 (32 的 2 的幂次倍数)
中断源 0保留 (不存在)
优先级3bit (0=禁用, 1=最低, 7=最高)
触发类型边沿/电平可配
Pending 位每源 1 bit, 中断到达时置位, claim 后清零

2.2.2 PLIC 仲裁逻辑

// PLIC 仲裁伪代码
function plic_arbitrate(context_id):
    // context_id: 每个 Hart 的 M-mode/S-mode 各一个 context
    best_id = 0
    best_priority = 0

    for src_id in 1..NUM_SOURCES:
        if pending[src_id] and enable[context_id][src_id]:
            if priority[src_id] > best_priority:
                best_priority = priority[src_id]
                best_id = src_id

    // Threshold 过滤
    if best_priority > threshold[context_id]:
        return {best_id, best_priority}  // MEIP/SEIP 信号置位
    else:
        return {0, 0}  // 无中断

// PLIC Claim/Complete
function plic_claim(context_id):
    {id, prio} = plic_arbitrate(context_id)
    pending[id] = 0  // 清除 pending
    return id  // 返回中断源 ID, 软件通过此 ID 识别中断

function plic_complete(context_id, src_id):
    // 中断处理完成后通知 PLIC, 允许同源新中断
    enable[context_id][src_id] = 1  // 重新使能

2.3.2 CLIC 优先级与抢占

// CLIC 优先级仲裁
function clic_arbitrate():
    // 遍历所有 CLIC 中断输入
    best_id = -1
    best_level = 0
    best_prv = M

    for irq_id in 0..CLIC_NUM_INTERRUPTS:
        if clic_pending[irq_id] and clic_enable[irq_id]:
            level = clic_level[irq_id]  // 中断级别 (8bit, 0~255)
            prv   = clic_prv[irq_id]    // 目标特权级 (M/S/U)

            // 级别 > 当前阈值 且 > 当前处理中断级别
            if level > mintthresh.th:
                if best_id == -1 or level > best_level:
                    best_id = irq_id
                    best_level = level
                    best_prv = prv

    if best_id >= 0:
        // 检查是否可抢占当前中断处理
        if best_level > mintstatus.mil:  // mil = 当前中断级别
            return {best_id, best_level, best_prv, PREEMPT}
        else:
            return {-1, 0, M, NO_PREEMPT}  // 不抢占
    return {-1, 0, M, NO_INTERRUPT}

// 硬件向量中断跳转
function clic_vector_jump(irq_id, prv):
    // 直接跳转到中断处理入口, 无需软件查表
    // vectored 模式: PC = mtvec.BASE + 4 * irq_id
    vector_addr = mtvec.BASE + 4 * irq_id
    return vector_addr

2.4 中断响应时序 D-013: 中断延时精化

2.4.1 16 cycle 预算分解

步骤延迟描述累计
1. 中断信号同步1 cycle外部异步信号 2-FF 同步到 clk_core1
2. PLIC 仲裁0 cycle组合逻辑 (已在 PLIC 内部完成)1
3. CLIC 仲裁1 cycleCLIC 优先级比较 + 抢占判断2
4. W 级中断检测0 cycle检查 mip & mie & mstatus.MIE (组合逻辑)2
5. 当前指令完成0~1 cycle等待 M/W 级排空 (顺序流水线)3
6. 上下文保存1 cyclemepc/mcause/mstatus 写入 CSR4
7. PC 跳转1 cycleF1 级: PC = mtvec (vectored: + 4*cause)5
8. 取指 (F2)1 cycle从 ITCM/I-Cache 取首条处理指令6
9. 解码 (D)1 cycle指令解码7
10. 发射 (I)1 cycle指令发射8
11. 执行 (E1)1 cycle首条中断处理指令执行9
预算分析: 硬件流水线部分约 9 cycle (步骤 1~11)。剩余 7 cycle 预算用于：
  • PLIC 仲裁延迟 (多中断源时): ~2~4 cycle
  • CLIC 向量查表 (如使用 shv): ~1~2 cycle
  • Cache miss (首条指令不在 I-Cache): 需额外 refill delay
最坏场景 (PLIC 多源仲裁 + I-Cache miss): 9 + 4 + 2 + refill ≈ 可能超过 16 cycle。
缓解策略: 中断向量表放置于 ITCM (零延迟), 避免 I-Cache miss。
审查约束响应 — D-013:
  • 预算分解: 16 cycle = 1 (同步) + 1 (CLIC) + 1 (排空) + 1 (CSR) + 1 (跳转) + 1 (F2) + 1 (D) + 1 (I) + 1 (E1) + 7 (PLIC+向量+裕量)
  • 关键路径: 中断向量在 ITCM 时 9 cycle 可达, 裕量 7 cycle
  • 风险场景: I-Cache miss 可能超时 → 建议 BSP 将中断向量固定于 ITCM
  • GEM5 验证: 中断延时测试用例 (perf/gem5-model/tests/) 可精化实际 cycle 数

2.5 嵌套中断规则

场景行为条件
同级中断不嵌套, 按 ID 仲裁CLIC level 相同
高优先级抢占嵌套: 当前中断暂停, 执行高优先级新 level > mintstatus.mil
MIE=0 期间不响应任何中断mstatus.MIE=0
手动允许抢占中断处理中设 MIE=1, 允许更高优先级软件显式操作

4.2 中断响应状态机

enum IrqState { IDLE, DETECT, DRAIN, CONTEXT_SAVE, JUMP, VECTOR_FETCH }

function irq_response_fsm():
    state = IDLE
    while true:
        // IDLE: 等待中断
        if interrupt_pending and current_instruction_committed:
            state = DETECT

        // DETECT: 确认最高优先级中断
        if state == DETECT:
            {irq_id, irq_level} = select_highest_priority()
            state = DRAIN

        // DRAIN: 排空流水线 (M/W 级)
        if state == DRAIN:
            pipeline_drain()  // 1~2 cycle
            state = CONTEXT_SAVE

        // CONTEXT_SAVE: 保存上下文到 CSR
        if state == CONTEXT_SAVE:
            mepc = current_pc + 4
            mcause = {1'b1, irq_id}  // Interrupt=1
            mstatus.MPIE = mstatus.MIE
            mstatus.MPP = current_privilege
            mstatus.MIE = 0
            state = JUMP

        // JUMP: 跳转到中断处理入口
        if state == JUMP:
            if mtvec.MODE == VECTORED:
                PC = mtvec.BASE + 4 * mcause.code
            else:
                PC = mtvec.BASE  // Direct mode
            state = VECTOR_FETCH

        // VECTOR_FETCH: 取首条处理指令
        if state == VECTOR_FETCH:
            // F1→F2→D→I→E1: 5 cycle 到首条执行
            state = IDLE  // 中断处理开始, 返回 IDLE
