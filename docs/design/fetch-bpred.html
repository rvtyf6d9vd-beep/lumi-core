1. 概述与范围

1.1 模块职责

Fetch & Branch Predictor 模块覆盖 Lumi-Core 8 级流水线的 F1 (Fetch1) 和 F2 (Fetch2) 两级，负责：

  • F1 级：PC 生成、分支预测目标计算、BTB 查找、ICache 地址送出
  • F2 级：ICache 数据返回、指令预解码、压缩指令 (C 扩展) 展开、分支预测结果确认

1.2 范围边界

包含不包含
PC 生成与更新逻辑完整指令解码 (属于 decode-issue.html)
LTAGE 分支预测器 (12 张 TAGE 表 + Loop Predictor)分支执行与判断 (属于 execute-alu.html)
BTB (8192 entries) + RAS (32 entries)ICache 存储阵列 (属于 cache.html)
压缩指令展开 (C/Zce 扩展)取指地址翻译 (PMP/PMA 检查属于 pma-axi.html)

关联模块: decode-issue.html (C 展开→D 级) | writeback-csr.html (分支结果→W 级 CSR) | execute-alu.html (JALR 反馈) | cache.html (ICache 取指) | pma-axi.html (PMA/PMP 检查) | debug-trace.html (halt 取指控制)

2. 功能描述

2.1 F1 级功能

2.1.1 PC 生成

// F1 级 PC 生成伪代码
if (branch_redirect) {
    next_pc = branch_target;      // 分支预测命中 → 跳转目标
} else if (return_predict) {
    next_pc = ras_pop();          // RAS 预测返回地址
} else {
    next_pc = current_pc + fetch_width;  // 顺序取指
}
// 对齐到 2 字节边界 (支持 C 扩展)
next_pc = next_pc & ~0x1;

2.1.2 BTB 查找

参数规格
条目数8192 (SimpleBTB)
索引方式PC[14:3] (直接映射)
每条目存储tag (PC 高位) + target_addr + branch_type (conditional/unconditional/call/return)
查找延迟0 cycle (与 PC 生成并行)

2.1.3 RAS 操作

操作触发描述
PushBTB 识别为 CALL 指令压入返回地址 (PC + 指令长度)
PopBTB 识别为 RETURN 指令弹出预测返回地址
容量32 entries支持 32 层嵌套调用

2.2 F2 级功能

2.2.1 指令预解码与压缩展开

  • 从 ICache 返回的 128bit (16 bytes) 数据中识别指令边界
  • 检测 16bit 压缩指令 (C/Zce 扩展) 并展开为 32bit 内部表示
  • 每个 cycle 最多处理 6 条指令 (fetch_width = issue_width × 2)
  • 预解码输出: 指令类型 (branch/load/store/alu/csr/system) + 寄存器编号

2.2.2 LTAGE 分支预测器

LTAGE = TAGE (TAgged GEometric history length) + Loop Predictor，是 Lumi-Core 的核心性能组件。

2.2.2.1 TAGE 组件

参数规格描述
表数量12 张几何级数历史长度 (minHist=4, maxHist=640)
总容量256 Kbits分布到 12 张表 + bimodal 基础表
Tag 宽度8~12 bit (按表不同)长历史表使用更宽 tag
计数器宽度3 bit (saturating)预测方向: ≥4 taken, <4 not-taken
有用性计数器2 bit (useful)替换策略依据
D-012 审查约束: 640 bit 全局历史寄存器 (GHR) 远超 RV32 的 32 bit 地址空间，必须通过 hash 压缩。

2.2.2.2 GHR 折叠策略 D-012

640 bit GHR 不能直接与 PC 进行 XOR 运算 (位宽不匹配)。采用 Folded History 策略：

// 32-bit Folded History 计算
function compute_folded_history(ghr[639:0], table_idx):
    hist_len = geometric_length(table_idx)  // 4, 6, 9, 14, 21, 32, 48, 72, 108, 162, 243, 364
    folded = 0x00000000  // 32-bit
    for i in range(hist_len):
        folded ^= (ghr[i] << (i % 32))
    return folded

// Tag 计算
tag = hash(PC[31:2], folded_history)  // XOR + rotate
// Index 计算
index = hash(PC[31:2], folded_history) % table_size

2.2.2.3 12 表 Tag 比较流水化 D-012

12 张 TAGE 表的 tag 比较如果全部并行，组合逻辑面积和延迟过大。采用分组流水化方案：

阶段比较表组延迟
F1 前半 (Phase A)Table 0~5 (短历史) + bimodal 基础表~40ps
F1 后半 (Phase B)Table 6~11 (长历史)~40ps
F1 仲裁选择最长历史命中的表作为预测结果~20ps

总延迟约 100ps，在 F1 级 125ps 预算 (1GHz @ TSMC 28HPC) 内。

2.2.2.4 Loop Predictor

参数规格
表条目128 entries
记录内容PC tag + 循环迭代计数 + 当前计数 + 置信度
预测规则当迭代计数稳定时, 预测循环最后一次为 not-taken

2.2.2.5 Loop Predictor 与 TAGE 优先级仲裁 D-012

// 优先级仲裁伪代码
if (loop_predictor.confident && loop_predictor.count_stable):
    prediction = loop_predictor.prediction  // Loop Predictor 优先 (对紧密循环近乎零误判)
elif (tage.longest_hit_table.useful >= 2):
    prediction = tage.longest_hit_table.prediction  // TAGE 高置信度
elif (tage.any_hit):
    prediction = tage.longest_hit_table.prediction  // TAGE 任意命中
else:
    prediction = bimodal.prediction  // 退回 bimodal 基础预测

2.3 状态机

Fetch FSM:
  IDLE  --(reset deassert)--> FETCHING
  FETCHING --(branch redirect)--> REDIRECT
  REDIRECT --(new PC ready, 0 cycle)--> FETCHING
  FETCHING --(sleep request)--> SLEEPING
  SLEEPING --(interrupt/wakeup)--> FETCHING

4. 数据通路

4.1 F1 级数据流

PC_gen ──→ BTB_lookup (并行) ──→ Branch_decision
       └─→ LTAGE_predict (并行)      │
                                      ↓
              if (predict taken && BTB hit):
                  next_PC = BTB_target
              elif (CALL):
                  RAS.push(PC + inst_len)
                  next_PC = BTB_target
              elif (RETURN):
                  next_PC = RAS.pop()
              else:
                  next_PC = PC + fetch_width

4.2 F2 级数据流

ICache_data[127:0] ──→ Boundary_detect ──→ Compress_expand ──→ Pre_decode ──→ Fetch_Buffer
                                              │                      │
                                              └── inst_type          └── reg_nums
                                              └── inst_valid         └── pred_info

4.3 分支误预测恢复

步骤cycle操作
1. 检测误预测E1 级Execute 级比较实际分支方向与预测方向
2. 发送 redirectE1 级branch_redirect_valid=1, branch_redirect_pc=正确目标
3. Flush 流水线E1+1清空 F2/D/I 级的错误指令
4. 新 PC 取指E1+1F1 使用 redirect PC 重新开始取指 (backwardDelay=0)

4.4 GHR 更新与回滚

GHR 在 F1 级推测性更新 (speculative update)，误预测时需回滚：

  • 推测更新: F1 预测时立即将预测方向移入 GHR (移位 + 插入)
  • 回滚: 误预测时，从 Execute 级保存的 GHR 快照恢复
  • 确认更新: 分支正确提交后，更新 TAGE 表计数器
