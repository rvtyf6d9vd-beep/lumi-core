Microarchitecture Pre-Design Review Report

1. 流水线架构审查

1.1 流水线深度与阶段划分

审查 8 级流水线 (F1→F2→D→I→E1→E2→M→W) 的阶段划分合理性，对比 GEM5 MinorCPU 实际建模。

Spec 阶段Spec 功能GEM5 对应对齐状态
F1 (Fetch1)PC 生成、分支预测、ICache 地址送出Fetch1对齐
F2 (Fetch2)ICache 返回、预解码、压缩指令展开Fetch2对齐
D (Decode)完整解码、寄存器读取、操作数准备Decode1 + Decode2命名差异
I (Issue)指令发射到 FU、数据旁路、发射仲裁Issue对齐
E1 + E2 (Execute)ALU 运算、地址计算、分支判断、MUL/DIVExecute (单级)分组差异
M (Memory)DCache/TCM 访问、Load 返回、Store 提交Memory对齐
W (Writeback)结果写回、异常/中断检测与提交Writeback对齐
发现 D-001 (中): Spec 将 Decode 列为 1 级，但 GEM5 MinorCPU 实际建模为 Decode1 (预解码) + Decode2 (完整解码+寄存器读取) 两级。Spec 将 Execute 列为 E1+E2 两级，但 GEM5 只有单个 Execute 级。两者总级数均为 8 级，但阶段分组不同。建议: 在设计文档中明确说明 spec 的 D 级对应 GEM5 的 Decode1+Decode2，spec 的 E1+E2 对应 GEM5 的 Execute 级。RTL 设计应遵循 spec 定义。

1.2 功能单元 (FU) 配比审查

对比架构说明书 §4.3 定义的 FU 配比与 GEM5 实际配置：

FU 类型Spec §4.3 定义GEM5 实际配置差异说明
ALU3 (opLat=1)4 (LumiIntFU, opLat=1)Phase 10 升级为 4x，减少 FU 竞争
MUL1 (opLat=3, 与 DIV 共享)1 (LumiIntMulFU, opLat=1)Phase 7 降低 opLat 为 1 (流水线化乘法器)
DIV(与 MUL 共享)1 (MinorDefaultIntDivFU, opLat=9)GEM5 中 MUL 和 DIV 是独立 FU
LSU1 (Load 1cycle TCM)2 (MinorDefaultMemFU, triple-issue)Phase 11 添加第二个 MEM FU (仅三发射)
FPU1 (FADD 4c / FMUL 4c)1 (MinorDefaultFloatSimdFU, opLat=6)延迟参数有差异，需在 FPU 文档中细化
Branch1 (1cycle)无独立 Branch FUGEM5 中分支通过 ALU 执行
MISC未定义1 (MinorDefaultMiscFU, opLat=1)系统指令 (ECALL/EBREAK/FENCE 等)
总计7 FU10 FU (三发射)
发现 D-002 (关键): Spec §4.3 的 FU 配比严重落后于 GEM5 优化模型。GEM5 已通过 Phase 7/10/11 优化将 CoreMark 从 3.05 提升到 6.97 CM/MHz。用户决策: 更新架构说明书 §4.3 以反映 GEM5 优化后的配置 (4xALU + 1xMUL + 1xDIV + 2xLSU + 1xFPU + 1xMISC)。

1.3 分支预测器审查

参数Spec §4.4GEM5 配置状态
预测器类型LTAGE (TAGE + Loop)LTAGE一致
TAGE 表数量12 张12 张 (gem5 默认)一致
历史长度minHist=4, maxHist=640gem5 默认一致
BTB 条目8192SimpleBTB(numEntries=8192)一致
RAS 条目32ReturnAddrStack(numEntries=32)一致
误判率<1%RV32 CoreMark: 1.19%略超

发现 D-003 (低): Spec 声称误判率 <1%，但 GEM5 RV32 CoreMark 实测为 1.19%。这是 CoreMark 的特定表现 (紧密循环多)，其他工作负载可能更低。建议将 spec 改为"典型工作负载误判率 ≤2%，CoreMark 紧密循环 ≤1.5%"，更符合实测数据。

1.4 旁路网络审查

旁路路径Spec 定义GEM5 实现状态
E1→D有有 (Execute→Decode forward)一致
E2→D有同上 (Execute 单级)一致
M→D有有 (Memory→Decode forward)一致
W→D未定义有 (Writeback→Decode)spec 遗漏
FPU→D未定义有 (通过通用 forward path)spec 遗漏
发现 D-004 (中): Spec §4.6.1 只列出 E1→D, E2→D, M→D 三条旁路路径，遗漏了 W→D 和 FPU 结果旁路。GEM5 MinorCPU 的 forward path 实际覆盖所有前序级到 Decode。建议: 在 spec §4.6.1 补充 W→D 旁路和 FPU 结果旁路路径说明。

1.5 冒险处理审查

冒险类型Spec 策略GEM5 实现状态
Load-Use RAW1 cycle stallexecuteLoadAssumedLatency=1 (推测命中)差异
MUL/DIV 多周期MUL 3cycle, DIV 迭代MUL opLat=1 (流水线), DIV opLat=9差异
Vector 多周期未定义GEM5 未建模 Vector FU缺失
FU 竞争 stall发射仲裁 + stallskip-stalled 机制 + FU 排队一致
发现 D-005 (中): GEM5 使用 executeLoadAssumedLatency=1 的推测性 L1D 命中预测机制 (Phase 9 优化贡献 +12.4% 性能)，但 Spec 未记录此机制。建议: 在 spec §4.6.1 补充"推测性 Load 延迟预测"机制说明，标注为可选优化。
发现 D-006 (关键): Vector 扩展的流水线交互完全未定义。Vector 指令可能需要多周期执行，与顺序流水线的同步机制 (stall/skip) 未在 spec 中描述。这是 M2-S1 设计文档必须解决的关键接口问题。

3. 模块间接口审查

3.1 接口审查结论

接口审查结论关键发现优先级
Core ↔ Memory需细化TCM 0/1拍 vs Cache 1+拍 的统一处理方式未定义。GEM5 使用推测命中 (latency=1) 但 miss 惩罚未说明高
Core ↔ CSR清晰CSR 读在 D 级，写在 W 级，旁路覆盖。spec §4.7.4 已定义低
Core ↔ Interrupt需细化≤16cycle 预算分配未明确。CLIC 仲裁延迟占几拍？pipeline flush 占几拍？高
Core ↔ Debug缺失Debug halt 在哪个流水线级生效？Resume 后流水线如何重新填充？完全未定义关键
Core ↔ FPU/Vector缺失FPU 多周期指令与流水线同步机制未定义。Vector 长指令是否需要独立 issue 机制？关键
Memory ↔ Bus清晰Cache miss 时 AXI 请求由 Cache 控制器发起。TCM DMA 从接口与主接口通过仲裁器分离低
Interrupt ↔ Bus清晰PLIC 寄存器通过 AXI MMIO 访问，地址映射在 spec §7.6 定义低
multicore ↔ Core缺失锁步比较器接入哪个流水线级？比较哪些信号 (PC? 寄存器? 内存写?)？完全未定义关键

3.2 关键接口定义需求

发现 D-007 (关键): 三个关键接口在 spec 中完全缺失，必须在 M2-S1 设计文档中定义：
  1. Debug halt/resume 流水线交互 — 建议在 debug-trace.html 中定义 halt 在 W 级生效 (指令提交后)，Resume 后从 halt PC 重新取指
  2. FPU/Vector 多周期同步 — 建议在 fpu-vector.html 中定义 FPU busy 信号回传 Issue 级，期间同类型 FU 指令 stall
  3. 锁步比较器接入点 — 建议在 multicore-lockstep.html 中定义比较在 W 级 (写回前)，比较 PC + 目标寄存器号 + 写入数据

4.2 性能基线对齐

基准Spec 目标GEM5 RV32 实测GEM5 RV64 实测状态
CoreMark≥5.5 CM/MHz6.97 CM/MHz6.22 CM/MHz达成
Dhrystone≥3.8 DMIPS/MHz1.65 DMIPS/MHz1.48 DMIPS/MHz未达成
中断延时≤16 cycledeferred (RISK-001)--待验证
发现 D-008 (关键): Dhrystone 目标 3.8 DMIPS/MHz 与 GEM5 实测 1.65 DMIPS/MHz 差距巨大 (2.3x)。对于顺序执行三发射处理器，3.8 DMIPS/MHz 是非常激进的目标 (ARM Cortex-R82 公开数据约 2.5 DMIPS/MHz)。建议: 将 Dhrystone 目标下调为 ≥2.0 DMIPS/MHz (仍优于典型双发射)，或在 PRD 中标注为"Stretch Goal"。需用户后续决策。

4.3 GEM5 模型已知限制

  • 中断延时: FS 模式 4/4 PASS (PMP/CLIC/PLIC/Smrnmi)，但 CLIC 自动接线需 gem5 重编译 (RISK-001/002)
  • Smepmp/Smrnmi: gem5 不支持这两个扩展的 CSR 访问 (ERR-004/005)
  • Vector: gem5 MinorCPU 未建模 Vector FU，Vector 指令行为未知
  • AtomicSimpleCPU bug: mepc+mret trap handler recovery 损坏 (ERR-007/RISK-005)
