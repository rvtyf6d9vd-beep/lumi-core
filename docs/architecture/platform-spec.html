1. 概述与范围

1.1 文档目的

本文档定义 Lumi-Core RISC-V 处理器 IP 的平台级架构，作为项目全局的技术基线。承上承接 PRD 产品需求，启下指导各模块的详细设计文档。所有章节内容均从 PRD product-requirements.html 需求条目追溯而来。

1.2 产品定位

Lumi-Core 是一款面向汽车电子、工业控制和嵌入式消费电子的高性能安全嵌入式 RISC-V 处理器 IP。架构定位为 RV32 顺序执行超标量处理器，对标 ARM Cortex-R52/M7 级别性能，同时具备完全可定制的开放架构优势。

1.3 设计约束 HARD CONSTRAINTS

  • 顺序执行 (In-Order)：禁止乱序执行，保证确定性和可分析性
  • 发射宽度 ≤ 3：最大三发射，不支持更宽配置
  • 流水线深度固定 8 级：不加深流水线
  • XLEN = 32：仅 RV32 架构
  • 物理地址 32bit：Little-Endian 字节序

4. 微架构定义 (Micro-architecture)

4.1 流水线概述

Lumi-Core 采用 8 级顺序执行超标量流水线 LUMI-PIPE-001，设计目标是在 RV32 架构上达到 CoreMark ≥5.5 CM/MHz LUMI-IPC-001。

性能基线 (gem5 MinorCPU 实测): CoreMark Triple 5.54 CM/MHz (目标 5.5 已达成), IPC 约 1.80。编译器: GCC -O2 -funroll-loops。

4.4 分支预测

  • 预测器：LTAGE (TAGE + Loop Predictor)，12 张 TAGE 表，minHist=4, maxHist=640, 总量 256Kbits
  • BTB：SimpleBTB, 8192 条目
  • RAS：Return Address Stack, 32 条目，支持函数调用快速返回
  • 误判率：TournamentBP 2.39% → LTAGE 显著降低，RV32 CoreMark 实测 1.19%，Loop Predictor 对紧密循环近乎零误判
  • 误判率目标：典型工作负载 ≤2%，CoreMark 紧密循环 ≤1.5% (v1.0.2 更新，基于 GEM5 实测数据)
  • fetch1ToFetch2BackwardDelay = 0（已优化，消除分支惩罚）

4.5 发射策略

每个周期最多发射 3 条指令（Triple-issue），遵循以下规则：

  • 指令按程序顺序发射到可用 FU
  • 同一 FU 类型不可超额分配（如 3 条 ALU 指令可同时发射，但 4 条不可）
  • Load/Store 指令独占 LSU
  • 当前主瓶颈：Load-Use stall 与 Cache miss 延迟 (Phase 10 后 ALU FU 已扩展至 4 个，竞争不再是主瓶颈)
架构约束: 明确禁止乱序执行、超过三发射、更深流水线。这些是 Lumi-Core 的硬性设计约束，不可通过配置突破。

4.6 流水线冒险处理 (Pipeline Hazards)

4.6.1 数据冒险 (Data Hazards)

冒险类型处理策略代价
RAW (读后写)全程数据旁路 (Forwarding/Bypass) 网络，E1→D, E2→D, M→D, W→D, FPU→D0 cycle (旁路命中时)
Load-Use RAWLoad 数据在 M 级才可用，若后指令 D 级需要，插入 1 个 stall cycle。GEM5 使用推测性 Load 延迟预测 (executeLoadAssumedLatency=1)，假设 L1D 命中，miss 时回滚1 cycle stall (推测命中时 0 cycle)
WAW (写后写)顺序提交保证 WAW 不发生0
WAR (写后读)顺序执行自然消除 WAR0

4.6.2 控制冒险 (Control Hazards)

  • 分支解析：在 E1 级完成分支判断，误预测时 flush F2/D/I 级流水线
  • 误预测惩罚：3 cycle flush (F2 + D + I 级)
  • LTAGE 优化：典型误判率 <1%，Loop Predictor 对紧密循环近乎零误判
  • JAL/JALR：JAL 在 F1 级即可解析目标，JALR 需等 E1 级寄存器值
  • JALR 解析延迟：2 cycle (D 级读 rs1 + E1 级算地址) + 1 cycle (branch feedback)，总计 3 cycle 误预测惩罚 (v1.0.3 补充, D-010)

4.6.3 结构冒险 (Structural Hazards)

  • FU 冲突：同一周期内同类 FU 请求超过可用数量时，发射级仲裁并 stall。GEM5 使用 skip-stalled 机制 (skip-stalled=True)，允许非冲突指令越过 stall 指令继续发射。skip-stalled 不破坏程序顺序 (in-order commit)：被跳过的指令仍在队列中等待，后续指令仅在同一 FU 不冲突时才越过发射，所有指令仍按程序顺序提交 (v1.0.3 补充, D-015)
  • ALU 充足：4x ALU 已满足三发射需求，ALU 竞争不再是瓶颈 (Phase 10 优化后)
  • LSU 双通道：三发射模式下 2x LSU，同周期最多允许 2 条 Load/Store 指令 (Phase 11 优化后)
  • MUL 流水线化：乘法 1 cycle (opLat=1)，流水线化，不阻塞后续指令 (Phase 7 优化后)
  • DIV 多周期：除法 9 cycle (opLat=9, 迭代式)，期间 DIV FU 不可用但其他 FU 可继续执行

4.7.1 整数寄存器堆 (Integer Register File)

端口压力分析 (v1.0.3, D-009): 三发射模式下整数寄存器堆 3R/2W 端口的最坏情况分析：
• 读端口需求：当 3 条指令同时需要读取 6 个不同源寄存器时 (如 2 条 ALU + 1 条 Load)，3R 端口不足，需分 2 个 cycle 完成寄存器读取

• 仲裁规则：当读端口不足时，发射级 stall 第三条指令直到前 2 条完成 D 级读；优先级按程序顺序 (先发射的指令优先占用端口)
• 实际影响：典型工作负载下端口冲突概率低 (大部分指令仅需 1~2 个源寄存器)，极端场景 (密集三发射 + 不同寄存器) 才会触发 stall

设计决策:
  • CLIC ≠ 单核：CLIC 是 per-hart ISA 扩展，不提供跨核路由
  • PLIC 是平台级 IP（非 ISA 扩展），提供多核有线中断分发
  • 两者互补而非替代
  • 不选择 AIA（APLIC+IMSIC）：面积大、生态较新，不适合嵌入式实时场景
  • RT-Europa 合规：ISA 扩展层选 CLIC 即满足 Profile 要求

5.2 中断信号路径

外部中断源 → PLIC（仲裁/路由）→ MEIP/SEIP 信号 → CLIC（每 Hart）→ Hart 核心

5.3 CLIC (Core-Local Interrupt Controller)

5.3.1 功能概述

  • 中断数范围：1 ~ 240（可配置，默认 16）
  • per-hart 实例：每个 Hart 独立 CLIC 实例
  • 中断类型：软件中断、定时器中断、外部本地中断
  • 优先级仲裁：多级优先级，支持抢占
  • 硬件向量中断：直接跳转到中断处理入口，无需软件查表

5.3.2 CLIC CSR 空间

CSR地址描述
mintstatus0xFB1中断状态 (当前中断级/子级/处理状态)
mintthresh0x347中断阈值
mclick0xFB0CLIC 中断 claim/complete

5.3.3 性能要求

快速中断延时 ≤16 时钟周期 LUMI-SYS-003：从中断信号到第一条中断处理指令执行。

5.4 PLIC (Platform-Level Interrupt Controller)

5.4.1 功能概述

  • 中断数范围：32 ~ 960（必须为 32 的 2 的幂次倍数，默认 32）
  • 优先级：7 级优先级（0=禁用, 1=最低, 7=最高）
  • 路由：支持多 Hart 路由，通过 enable 位选择目标
  • Claim/Complete：标准 PLIC 协议

5.4.2 PLIC 寄存器映射

偏移名称描述
0x000000priority中断源优先级 (每个源 4 字节)
0x001000pending中断挂起位
0x002000enable中断使能 (per-context)
0x200000threshold/claim阈值与 claim (per-context)

5.5 中断信号流向 (Interrupt Signal Flow)

完整的中断传递路径如下：

 外部设备 (UART, Timer, GPIO, ...)
      │ IRQ[0..N-1]
      ▼
 ┌────────────┐
 │   PLIC     │  ● 优先级仲裁 (7 级)
 │ (平台级) │  ● 多 Hart 路由 (enable 位)
 │            │  ● Threshold 过滤
 └────┬─────┘
      │ MEIP / SEIP (per-hart 信号)
      ▼
 ┌────────────┐
 │   CLIC     │  ● Per-hart 实例
 │ (ISA 层) │  ● 本地中断 + PLIC 传入的外部中断
 │            │  ● 硬件向量中断 (mtvec vectored)
 │            │  ● 优先级抢占
 └────┬─────┘
      │ 中断请求 (interrupt pending + enabled)
      ▼
 ┌────────────┐

 └────────────┘

5.5.1 中断响应时序

步骤流水线级动作
1. 中断检测W (Writeback)检查 mip & mie，判断是否有 pending+enabled 中断
2. 当前指令完成W当前指令提交，不取新指令
3. 上下文保存--mepc=pc+4, mcause=interrupt_id, mstatus.MIE→mstatus.MPIE, mstatus.MIE=0
4. 跳转F1PC = mtvec.BASE + ( vectored ? 4*mcause : 0 )
5. 首条处理指令F1开始取中断处理代码
中断延时目标: 从信号到达到第一条中断处理指令执行 ≤16 时钟周期 LUMI-SYS-003。其中步骤 3~5 硬件流水线约 5~8 cycle，剩余预算用于 CLIC 仲裁延迟。

5.5.2 嵌套中断规则

  • 同优先级中断不嵌套，按 ID 顺序仲裁
  • 高优先级中断可抢占低优先级 (CLIC 抢占模式)
  • 中断处理中 mstatus.MIE=0，禁止同级/低级中断
  • 软件可通过在中断处理中手动设 mstatus.MIE=1 允许高优先级抢占

5.6 异常处理架构 (Exception Handling)

7.8 PMA 检查器 (Physical Memory Attributes)

7.8.2 PMA 配置方式

  • 默认 PMA 映射：按 §7.6 全局地址映射静态编码

    • 0x8000_0000~0xBFFF_FFFF (DDR)：cacheable=true, executable=true, main_memory=true
    • 0xC000_0000~0xF000_FFFF (MMIO)：cacheable=false, executable=false, io=true
  • 可配 PMA（选项）：提供 cfgaddr (0xFD0) / cfgdata (0xFD1) CSR 用于运行时 PMA 定制（高阶客户使用）

8.4.1 NMI 进入流程

 NMI 信号到达
      │
      ▼
 1. mnepc = 当前 PC (被中断指令)
 2. mncause = NMI 原因码

8.5 信息安全机制

8.5.1 安全调试策略 LUMI-SEC-003

调试安全机制基于 RISC-V Debug V1.0.0 规范，提供多级安全调试控制：

  • Debug Lock/Unlock：通过 dmcontrol CSR 或外部 debug_lock 引脚控制调试端口使能状态
  • 安全等级配置：支持三级调试安全等级 — 完全锁定 / 仅断点 / 完全开放
  • 锁定保护：锁定状态下禁止调试器连接、Abstract Command 执行和 System Bus Access
  • 解锁机制：Lock 状态不可通过软件绕过，需硬件复位或密钥认证解锁

8.5.2 PMP 安全隔离 LUMI-SEC-004

基于 PMP + Smepmp 的安全隔离机制（引用 §3.2）：

  • Smepmp 强制生效：M-mode 下 PMP 配置不可被 S/U-mode 代码修改
  • PMP 配置锁定：锁定后非特权代码无法篡改内存访问权限
  • IP 隔离能力边界：基于 PMP 的内存访问控制，不含 MMU-based 安全域隔离（如 TrustZone 等价机制）

8.5.3 总线完整性保护 LUMI-SEC-005

从信息安全视角补充总线保护（引用 §7.5，对应功能安全 LUMI-SAFE-003）：

  • Parity 校验：AXI 主总线和 FFP 外设总线支持数据 Parity 校验
  • 异常处理：检测到完整性错误时触发 bus_error_exception
  • 策略可配：异常处理策略可配置（异常中断 / NMI / 系统复位）

8.6 多核架构 (Multi-core Architecture)

8.6.1 多核互联拓扑 LUMI-MCORE-001

  • 执行模型：两核执行相同指令流，每个指令周期后比较结果
  • 比较粒度：指令级比较（PC、寄存器值、内存写入数据）
  • 错误检测：检测到不一致时触发 Lock-Step Error 异常
  • 错误处理策略：可配置为 NMI / 系统复位 / 日志记录
  • 时钟域：锁步模式下两核必须共享同一时钟域
  • 配置参数：通过核数配置 (核数≥2) + 锁步使能位启用

8.7 可测试性设计 (DFT)

8.7.1 Scan Chain 扫描链 LUMI-DFT-001

  • 全扫描链插入：RTL 设计预留 DFT 插入点，支持扫描链自动插入
  • 扫描覆盖率：目标 ≥98% stuck-at fault 覆盖率
  • ATPG 支持：支持主流 ATPG 工具生成测试向量
  • 扫描链组织：支持多扫描链分组，减少测试时间

8.7.2 MBIST 内存内建自测试 LUMI-DFT-002

  • 覆盖范围：TCM (ITCM/DTCM0/DTCM1) 和 L1 Cache (I-Cache/D-Cache)
  • 测试算法：March C- 或等价算法，覆盖 stuck-at / transition / coupling 故障
  • ECC 联调：MBIST 与 ECC 保护协同工作，MBIST 期间可禁用 ECC 检查

8.7.3 JTAG 边界扫描 LUMI-DFT-003

  • 标准合规：IEEE 1149.1 边界扫描，与调试 JTAG 复用物理接口
  • BSDL 文件：交付完整 BSDL 描述文件
  • 支持指令：EXTEST / INTEST / SAMPLE / PRELOAD / BYPASS

8.8 物理设计约束 (Physical Design Constraints)

8.8.1 综合约束 LUMI-TIMING-001

  • SDC 文件：交付完整时序约束文件 (SDC 格式)，覆盖全部工作模式
  • 角落条件：覆盖 ss/tt/ff x 温度 x 电压的全部 PVT 组合
  • 目标工艺：TSMC 28HPC+ 作为基准工艺节点
  • 多角多模 (MCMM)：支持多场景约束合并 (Functional / Scan / MBIST / Sleep)

8.8.2 时序目标 LUMI-TIMING-002

  • ss corner：0.81V, 125°C 下时序收敛，WNS ≥ 0
  • ff corner：1.1V, -40°C 下无 hold violation
  • 关键路径：预计关键路径在 ALU/分支预测/Cache Tag 之间

8.9 时钟与复位架构 (Clock & Reset Architecture)

8.9.1 时钟域划分 LUMI-CLK-001

Lumi-Core 采用多时钟域设计，以平衡性能、功耗和外部集成灵活性：

时钟域信号名驱动源默认频率说明
CPU Core Clockclk_core外部 PLL/SoC同主频 (≥1.0GHz)驱动 Hart 核心、FPU、向量单元
Bus Clockclk_bus外部 SoC同主频或分频驱动 AXI 总线接口、PLIC、CLIC MMIO
TCM Clockclk_tcm同 clk_core同主频驱动 TCM 存储阵列（与 Core 同步访问）

8.9.2 Clock Gating 策略 LUMI-CLK-002

采用条件时钟门控 (Clock Gating) 降低动态功耗：

  • 细粒度门控：每个流水线级寄存器使用 ICG (Integrated Clock Gating) cell，门控使能信号由空闲检测逻辑产生
  • WFI 触发：WFI 指令执行后关闭 Core Clock 域，仅保留中断检测逻辑运行（引用 §8.2）
  • 手动控制：SoC 可通过 core_clk_gate_req 信号强制门控 clk_core
  • PLL Bypass：深度睡眠时 SoC 可关闭 PLL，由 ext_clk 维持寄存器状态
  • 门控恢复延时：WFI 唤醒后首条指令到执行 ≤8 cycle（门控使能 + 流水线填充）

8.9.3 复位域划分

复位信号按域分组，确保最小化复位影响范围：

复位域信号名触发源影响范围

Bus Resetbus_rst_n由 reset_n 派生仅 AXI 接口、PLIC、CLIC 寄存器复位

8.9.4 UPF/CPF 电源约束接口 LUMI-PWR-004

交付标准 UPF (Unified Power Format) 2.1 电源约束文件，定义电源域、隔离策略和保持寄存器：

  • 电源域：默认单电源域 PD_CORE，可扩展多域（PD_CACHE、PD_TCM）用于高级功耗管理
  • 隔离策略：跨电源域信号使用 isolation cell（默认 clamp 0），由 iso_en 信号控制
  • 状态保持：关键 CSR（如 mstatus, mepc, mtvec）使用 retention register 保持状态 LUMI-PWR-005
  • 电平转换：跨电压域信号使用 level shifter
  • UPF 交付物：docs/design/power/lumi-core.upf，含完整 power intent 描述
