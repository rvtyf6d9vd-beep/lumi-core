# ERR-013/014/015/016 修订修复计划

## 根因分析（修订版）

### ERR-013: FENCE 解码缺失 — 已修复
`lumi_decode_issue.sv` 已添加 `OP_FENCE` 常量和 decode case（FU_MISC, NOP-like）。此部分保留不变。

### ERR-015: FENCE 在 execute 中被误判为 ECALL（新发现，致命）
`lumi_execute.sv` L427 的 ECALL 检测：
```systemverilog
if (e1_inst[i].inst[31:20] == 12'h000 && e1_inst[i].funct3 == 3'b000)
```
FENCE 编码 `0x0000000f` 的 `inst[31:20]=12'h000` 且 `funct3=3'b000`，**完全匹配 ECALL 条件**，但缺少 opcode 校验。FENCE 触发 ECALL 异常 → `trap_taken=1` → 阻塞同 W 批次所有 `commit_valid`（包括 CSR）→ CSR + FENCE 均不提交。

### ERR-014: issued_mask 竞态 — 根因修正
原计划分析 PC 变化时清 mask 的思路正确，但实施中有三处致命缺陷：

1. **mask 仅在 stall_out 时记录**：当批次后半部分在非 stall 周期发射时（stall_out=0），这些指令的 mask 位**不被设置**。下一个 bubble 周期中，这些未 mask 的旧指令被**重新选择和发射**，导致流水线寄存器覆盖、指令丢失。

2. **batch_done_r 引入额外死周期**：批次完成后设置 batch_done_r=1 阻止所有发射，直到新批次 PC 到达。但 F2 管线延迟使新批次 PC 需 2 个周期才到达，造成每批次 2 个 bubble 而非 1 个。实测覆盖率从 65% 跌至 40%。

3. **F2 freeze (!dec_stall) 产生 RAW 误判**：dec_stall 期间冻结 F2 数据，但 fetch FSM 已冻结 PC（`pc_next = pc_reg`），F2 捕获的本来就是相同数据。F2 freeze 阻止了正常的 `f2_valid_r <= 0` 失效，在批次切换时造成数据不一致。

### ERR-016: JAL/JALR 检测使用 funct7 位（新发现）
`lumi_execute.sv` L366 用 `funct7[6]` 检测 JAL、`funct7[5]` 检测 JALR。但 JAL 的 `funct7=inst[31:25]=imm[20:14]`，JALR 的 `funct7=inst[31:25]=imm[11:5]`。小偏移前向跳转时这些位为 0 → JAL/JALR 被误当条件分支处理。

### 1-cycle F2 bubble（设计特征，非 bug）
F2 寄存器在时钟沿捕获 `pc_reg`（旧值），而 `pc_reg <= pc_next` 在同一沿生效。新 PC 的 ICache 数据在下一周期才组合到达。每个批次边界存在 1 个 bubble 周期，这是 6-wide fetch + 组合 ICache 旁路的固有延迟，不可消除。

```systemverilog
always_comb begin
    issued_mask_next = issued_mask_r;
    batch_pc_next    = batch_pc_r;

    if (flush) begin
        issued_mask_next = '0;
        batch_pc_next    = d_pc;
    end else if (d_valid && d_pc != batch_pc_r) begin
        // 新批次到达: 清除 mask
        issued_mask_next = '0;
        batch_pc_next    = d_pc;

    for (int s = 0; s < ISSUE_WIDTH; s++) begin
        if (issue_ready[s])
            issued_mask_next[issue_sel[s]] = 1'b1;
    end
end
```

## Task 4: 修复 ERR-016 — JAL/JALR 检测
**文件**: `src/rtl/core/lumi_execute.sv`

L366-383 的 FU_BRANCH 处理改用 opcode 区分：
```systemverilog
FU_BRANCH: begin
    is_branch[i] = 1'b1;

    // JAL: opcode=1101111
    if (e1_inst[i].inst[6:0] == 7'b1101111) begin
        branch_taken[i]  = 1'b1;
        branch_target[i] = e1_inst[i].pc + e1_inst[i].imm;
        e1_result[i]     = e1_inst[i].pc + 32'd4;
    end
    // JALR: opcode=1100111
    else if (e1_inst[i].inst[6:0] == 7'b1100111) begin
        branch_taken[i]  = 1'b1;
        branch_target[i] = (e1_rs1_data[i] + e1_inst[i].imm) & ~32'h1;
        e1_result[i]     = e1_inst[i].pc + 32'd4;
    end
    // 条件分支: opcode=1100011
    else begin
        branch_taken[i] = branch_evaluate(
            e1_inst[i].funct3, e1_rs1_data[i], e1_rs2_data[i]);
        branch_target[i] = e1_inst[i].pc + e1_inst[i].imm;
    end

    // 误预测检测 (保持不变)
    if (e1_branch_taken != branch_taken[i] ||
        (branch_taken[i] && e1_branch_target != branch_target[i])) begin
        e1_mispredict = 1'b1;
    end
    if (!e1_mispredict) begin
        e1_branch_taken  = branch_taken[i];
        e1_branch_target = branch_target[i];
    end
end
```

## 风险缓解

| 风险 | 缓解措施 |
|------|----------|
| mask always-record 在 bubble 周期阻止全部发射 | 正确行为: bubble 周期 mask=全 1, 无指令可选, stall_out=0, PC 前进 |
| JAL 前向跳转触发 mispredict flush | 预期行为: BTB 冷启动不预测, JAL 实际 taken → flush + redirect, 验证重定向路径 |
| FENCE 不触发异常后 CSR 仍不提交 | 检查 CSR 是否与 ECALL 在同 W 批次 (writeback trap_taken 阻塞) |
